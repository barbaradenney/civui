`<civ-repeater>`

"Add another" pattern for repeating form sections. Renders one summary card per item, with controls for adding, editing, and removing. Field names are automatically indexed in the inline / form-steps modes (e.g., `dependents[0].firstName`, `dependents[1].firstName`).

## Modes

### `inline` (default)

All rows and their fields are visible on the page simultaneously. The repeater captures its slotted children as a template and clones one copy per row. Best for simple items with 1–3 fields (phone numbers, email addresses, short text).

### `form-steps`

Rows render as compact summary cards. Clicking **Add** or **Edit** replaces the list with an in-place `civ-form-step` wizard — the user stays on the same page, the URL doesn't change. Best for items with 4–10 fields where page context matters (modal flows, sensitive flows, hosts without routing infrastructure).

### `route`

Rows render as summary cards driven by a host-owned `rows` array. **Add** and **Edit** are real `<a href>` links pointing at host-rendered pages. Browser back, deep linking, right-click "open in new tab", and save-and-resume all work naturally. The repeater never mutates `rows` — **Remove** fires `civ-repeater-remove` and the host updates its array.

Use route mode when each item is complex enough to deserve its own URL (8+ fields, multi-section items, items the user might want to bookmark and finish later). Slotted children are ignored in this mode; the form lives on the destination page.

| Item complexity | Mode | URL changes |
|-----------------|------|-------------|
| 1–3 simple fields | `inline` | No |
| 4–10 fields, stay on page | `form-steps` | No |
| 8+ fields, own page / save-and-resume | `route` | Yes |

## Default empty list

`min` defaults to `0`, so the repeater opens with **just the Add button** — the first row appears when the user clicks. Pass `min="1"` (or higher) to render that many rows up front. The minimum also acts as the floor for removal: rows can't be removed below `min`.

## Template definition (inline / form-steps)

The component captures its initial children as a clonable template. Each row is a clone with indexed names:

```html
<civ-repeater legend="Dependents" name="dependents" item-label="dependent" min="1" max="10">
  <civ-text-input label="First name" name="firstName"></civ-text-input>
  <civ-text-input label="Last name" name="lastName"></civ-text-input>
</civ-repeater>
```

This produces fields named `dependents[0].firstName`, `dependents[0].lastName`, etc.

## Route mode (controlled rows)

In route mode the repeater is **controlled** — the host owns the rows array. The repeater renders summary cards from `.rows` and emits `civ-repeater-remove` (with `{ index, id, row }`) when the user removes one. The host updates its own array and re-passes.

Summary text for each row is resolved through a fallback chain:

1. `rowSummary` function — `(row, index) => string`, full programmatic control (JS-only).
2. `summary-fields` attribute — comma-separated property names, joined with spaces. Works in HTML / Drupal / SSR contexts.
3. Falls back to `"{itemLabel} {index+1}"` when neither is set.

```html
<civ-repeater
  legend="Your dependents"
  item-label="dependent"
  mode="route"
  add-href="/dependents/new"
  edit-href-pattern="/dependents/{id}/edit"
  id-field="id"
  summary-fields="firstName,lastName"
  .rows=${this.dependents}
  @civ-repeater-remove=${(e) => this.removeDependent(e.detail.id)}
></civ-repeater>
```

**URL stability.** `edit-href-pattern` supports `{id}` (interpolated from `row[idField]`) and `{index}` (zero-based). Prefer `{id}` with stable identifiers — indexes shift when rows are removed, so bookmarks built from `{index}` will point at the wrong row after reorder/delete.

## Per-row heading

Every row renders a heading like "Dependent 1" as a visible group label (and the row's `aria-labelledby`). The heading element's tag is derived from `headingLevel` so the row sits one level deeper than the legend: when the legend isn't promoted to a heading the row uses `<h3>`; with `heading-level="3"` it's `<h4>`; clamped at `<h6>`.

The text comes from `repeaterItemLabel` (e.g. "dependent 1") and is capitalized in CSS — keep your `item-label` lowercase so it composes naturally with the Add button ("Add another dependent") and announcements.

## Programmatic API

```js
const repeater = document.querySelector('civ-repeater');

repeater.rowCount;        // Current number of rows (inline / form-steps)
repeater.addRow();        // Add a row (inline / form-steps)
repeater.removeRow(1);    // Remove the row at index 1 (inline / form-steps)

// Route mode is controlled — assign the new array on remove:
repeater.rows = repeater.rows.filter(r => r.id !== removedId);
```

## Adding confirmation to Remove

The default Remove button is one-click — appropriate for short scratch
lists where re-entering data is trivial. For lists where removed
content is harder to recover (entered dependent info, uploaded files,
typed addresses), insert a confirmation step.

CivUI's rule is **modal-only for destructive confirmations** — inline
two-click patterns are an accessibility hazard. See
[Destructive actions](/foundations/destructive-actions) for the
full rationale.

The pattern: listen for the cancelable `civ-repeater-before-remove`
event, call `preventDefault()` to abort the auto-remove, open your own
`civ-modal`, and on confirm re-call `removeRow(index, { skipConfirm:
true })`:

```ts
const repeater = document.querySelector('civ-repeater');
const modal = document.querySelector('#confirm-remove');

repeater.addEventListener('civ-repeater-before-remove', (e) => {
  e.preventDefault();
  const index = e.detail.index;
  modal.heading = `Remove ${repeater.itemLabel} ${index + 1}?`;
  modal.body = `This can't be undone.`;
  modal.onConfirm = () => repeater.removeRow(index, { skipConfirm: true });
  modal.open();
});
```

In **route mode**, the consumer already owns the rows array. There's
no separate before-remove event — wrap the modal logic around your
`civ-repeater-remove` handler, splicing the array only after confirm.

## Usage

```html
<!-- Inline mode (simple items) -->
<civ-repeater
  legend="Previous addresses"
  name="addresses"
  item-label="address"
  min="1"
  max="5"
>
  <civ-text-input label="Street address" name="street" required></civ-text-input>
  <civ-text-input label="City" name="city" required></civ-text-input>
</civ-repeater>

<!-- Form-steps mode (in-place wizard) -->
<civ-repeater
  legend="Your dependents"
  name="dependents"
  item-label="dependent"
  mode="form-steps"
  max="10"
>
  <div data-step-label="Name">
    <civ-name legend="Dependent's name" name="name" required></civ-name>
  </div>
  <div data-step-label="Relationship">
    <civ-relationship
      legend="Relationship details"
      name="rel"
      preset="dependent"
      hide-name
      required
    ></civ-relationship>
  </div>
</civ-repeater>

<!-- Route mode (separate page per item) -->
<civ-repeater
  legend="Your dependents"
  item-label="dependent"
  mode="route"
  add-href="/dependents/new"
  edit-href-pattern="/dependents/{id}/edit"
  id-field="id"
  summary-fields="firstName,lastName"
></civ-repeater>
<!--
  Host renders the form on /dependents/new and /dependents/{id}/edit,
  passes the rows array via `.rows=`, and listens for
  civ-repeater-remove to update its own state.
-->
```
