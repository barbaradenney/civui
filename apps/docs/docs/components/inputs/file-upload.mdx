### Password-Protected Files

Medical records (Kaiser, Mayo, VA Healthcare) and legal documents are commonly distributed as encrypted PDFs where the password is the recipient's date of birth or similar. Rather than rejecting the file or asking the user to strip the password externally, mark the file `locked` and the component renders an inline password input with show/hide reveal. On unlock the component fires `civ-file-unlock` with `{ index, name, file, password }`; the consumer's upload pipeline owns decryption (server-side via qpdf/pdftk, or client-side via pdf.js).

```ts
fileUpload.addEventListener('civ-change', async (e) => {
  // ... upload each file, server detects encryption ...
  if (serverResponse.status === 'password-required') {
    fileUpload.setFileStatus(index, 'locked');
  }
});

fileUpload.addEventListener('civ-file-unlock', async (e) => {
  const { index, file, password } = e.detail;
  const result = await retryUploadWithPassword(file, password);
  if (result.ok) {
    fileUpload.setFileStatus(index, 'success');
  } else {
    fileUpload.setFileStatus(index, 'locked', { error: 'Incorrect password. Try again.' });
  }
});
```

The password is cleared from the component's internal buffer immediately after dispatch — consumers must not log it, persist it, or send it over a non-HTTPS channel. The `<civ-text-input>` rendered for the password uses `type="password"`, `autocomplete="off"`, and `reveal-password` to support entering an unfamiliar DOB blind without errors.

### Duplicate Detection

Files matching by name + size + lastModified are rejected on add with a `{name} is already in the list` error. Removing then re-adding works as expected.

<StoryEmbed id="forms-inputs-file-upload--duplicate-detection" />

### Draft Restore (initial files)

Hydrate the list from a previous session by setting `el.initialFiles = [{ id, name, size, type?, url? }]`. Initial files render as already-uploaded; when `url` is set, the filename becomes a download link. Removing fires `civ-file-removed` with `{ isInitial: true, id }` so the consumer can issue server-side deletes. Form submission emits `${name}.kept-initial-ids` so the server can diff originally-issued IDs against what remains.

Two getters surface the draft-restore state for consumers that want to query it directly (in addition to listening for `civ-file-removed`):

- `el.keptInitialFileIds` — IDs of initial files still present in the list
- `el.removedInitialFileIds` — IDs of initial files removed since hydration

<StoryEmbed id="forms-inputs-file-upload--initial-files" />

### Reading the current files

Use `el.files` (returns `File[]`) to read the current selection programmatically. `el.value` is a comma-joined name string for display only — it does **not** carry file contents, so `JSON.stringify(form.values)` on a `civ-file-upload` field yields `"a.pdf, b.pdf"` rather than the file data. For form submission, the component manages its own `FormData` via `valueMode: 'file'` so the underlying `<civ-form>` does the right thing automatically; reach for `el.files` only when you need direct access from outside a form context.

## Upload Progress API

The component supports tracking upload progress via JavaScript:

```js
const upload = document.querySelector('civ-file-upload');

// Set file status
upload.setFileStatus(0, 'uploading', { progress: 50 });
upload.setFileStatus(0, 'success');
upload.setFileStatus(0, 'error', { error: 'Network timeout' });

// Get abort controller for cancellation
const controller = upload.getAbortController(0);
```

## Adding confirmation to Remove

The default Remove (✕) button next to each file is one-click. Real
.gov flows often involve uploads that took the user real time and
effort to produce (a 12 MB DD-214, an evidence packet) and an
accidental click loses that work. Insert a confirmation step.

CivUI's rule is **modal-only for destructive confirmations**. Inline
two-click patterns are an accessibility hazard. See
[Destructive actions](/foundations/destructive-actions) for the
full rationale.

The pattern: listen for the cancelable `civ-file-upload-before-remove`
event, call `preventDefault()` to abort the auto-remove, open your own
`civ-modal`, and on confirm re-call `removeFile(index, { skipConfirm:
true })`:

```ts
const upload = document.querySelector('civ-file-upload');
const modal = document.querySelector('#confirm-remove');

upload.addEventListener('civ-file-upload-before-remove', (e) => {
  e.preventDefault();
  const { index, name } = e.detail;
  modal.heading = `Remove ${name}?`;
  modal.body = `${name} will be removed from your upload. This can't be undone.`;
  modal.onConfirm = () => upload.removeFile(index, { skipConfirm: true });
  modal.open();
});
```

For server-resident files (the `initialFiles` set), `e.detail.id`
echoes the server identifier so you can issue a DELETE to the
backend in the same confirm handler.

## Usage

```html
<!-- Single file upload -->
<civ-file-upload
  label="Upload your DD214"
  name="dd214"
  accept=".pdf,.jpg,.png"
  max-size="10485760"
  hint="Upload a PDF or image of your DD214. Maximum file size: 10 MB."
  required
></civ-file-upload>

<!-- Multiple files with preview -->
<civ-file-upload
  label="Supporting documents"
  name="documents"
  accept=".pdf,.jpg,.png,.doc,.docx"
  multiple
  max-files="5"
  max-size="20971520"
  show-preview
></civ-file-upload>

<!-- Compact variant -->
<civ-file-upload
  label="Profile photo"
  name="photo"
  accept="image/*"
  variant="compact"
  max-size="5242880"
></civ-file-upload>
```
