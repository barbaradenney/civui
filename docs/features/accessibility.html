Accessibility First

CivUI is built from the ground up for Section 508 compliance and WCAG 2.1 AA conformance, ensuring government digital services are usable by everyone.

Light DOM Architecture

CivUI components render to the Light DOM instead of Shadow DOM. Where most web component libraries isolate their internals behind a shadow root, CivUI keeps everything in the real document tree. This architectural decision eliminates entire categories of accessibility barriers.

External Labels Work Naturally

The for/id relationship between labels and inputs works without workarounds. Assistive technologies see the connection natively.

ARIA IDREFs Cross Boundaries

Attributes like aria-describedby and aria-labelledby can reference any element in the document, not just those within a shadow root.

Straightforward Focus Management

Focus order follows the DOM naturally. No shadow root boundaries to manage, no delegatesFocus complexity, no unexpected focus traps.

Real DOM for Screen Readers

Screen readers see the actual document structure. No flattened tree ambiguity, no slot reassignment confusion, no double-announcement issues.

Native Form Participation with ElementInternals

CivUI form components use the ElementInternals API for native form participation. There are no hidden <input> elements injected into the DOM. Components work directly with the browser's form system.

Works with <form> and FormData

Component values appear in FormData automatically. Standard form submission, both traditional and via fetch, includes CivUI field values without extra JavaScript.

Constraint Validation API

Components integrate with the browser's built-in validation. Methods like checkValidity() and reportValidity() work as expected on CivUI elements.

formResetCallback()

When a form is reset, each CivUI component automatically restores its initial value. No manual event listeners needed.

formDisabledCallback()

Disabling a <fieldset> cascades the disabled state to all CivUI components inside it, matching native element behavior.

Government Compliance Checklist

Every CivUI component enforces these accessibility requirements by default. This is not optional configuration. These behaviors are built into the component architecture.

  • Every input has a visible label (or legend for group components)
  • Required fields render an asterisk and set aria-required="true"
  • Error messages use role="alert" for immediate screen reader announcement
  • Color is never the sole error indicator — errors use text and a border
  • All interactive elements display a visible focus ring via focus-visible
  • Labels use plain language: "Date of birth" not "DOB", "Social Security number" not "SSN"
  • Keyboard navigation throughout — roving tabindex for radio groups and segmented controls
  • Screen reader announcements queued via aria-live regions with polite and assertive priorities

Dark Mode

CivUI supports dark mode automatically via prefers-color-scheme: dark. The system respects the user's operating system preference with no toggle required.

The design token build pipeline validates that every light mode token has a corresponding dark mode counterpart. This parity validation runs at build time, ensuring no component is left unstyled in either mode. Focus ring colors, error states, and all semantic colors adapt to the active color scheme.

Density System

The density system provides dense, spacious, and fluid scale variants, applied via a data-civ-scale attribute on any parent element. Spacing and font sizes adjust per scale, supporting WCAG 1.4.4 (Resize Text) by ensuring content remains usable when users configure larger text sizes or when agencies need compact layouts for data-heavy screens.

<!-- Dense layout for data tables -->
<div data-civ-scale="dense">
  <civ-text-input label="Search" name="q"></civ-text-input>
</div>

<!-- Spacious layout for public-facing forms -->
<div data-civ-scale="spacious">
  <civ-text-input label="Full name" name="name" required></civ-text-input>
</div>

Consistent Rendering Order

Every CivUI form component renders its parts in the same predictable order. Screen reader users, keyboard users, and sighted users all encounter information in the same sequence. This consistency reduces cognitive load and makes forms easier to navigate.

1 Label
2 Hint
3 Error
4 Control
