# Timeline

<!-- contract-link:start -->

> **Contract reference:** [`civ-timeline`](pathname:///civui/storybook/?path=/docs/contract-civ-timeline--docs). Auto-generated structured contract for cross-platform implementation.

<!-- contract-link:end -->

`<civ-timeline>` + `<civ-timeline-item>`

A vertical timeline that lists dated events with an intent-colored rail connecting them. Generic over use cases — claim histories, audit logs, case progress, document workflows, version histories, project milestones, onboarding journeys, and any other "things that happened, in order" surface.

## When to use it

- A claim or case history page showing every status change over time.
- An audit log showing recent activity on an account (newest first).
- A document workflow page showing review / approval / publication steps.
- A version-history surface for releases, configuration changes, or content revisions.
- A submission progress page that walks the user through processing stages.

For a horizontal step indicator inside an active form, use [`civ-progress-steps`](../form/progress) instead — that's a forward-looking "where am I in this flow" component, not a backward-looking "what happened so far" log.

Still deciding between the timeline, process list, task list hub, and progress indicators? See [Choosing a Step Display](../../form-patterns/choosing-step-displays).

## Content model

Items are direct children. Order in the markup is the visual order — author oldest-first for chronological narrative (claim history) or newest-first for audit logs.

The rail (dot + connecting line) is rendered by each item, not the container. This keeps the rail aligned with the content even as items are added or removed dynamically.

:::caution Dynamic items

Avoid `${items.map(...)}` patterns that mutate the items array AFTER mount — the timeline uses `LightDomSlotMixin`, which captures children once in `connectedCallback`. Items added/removed by an outer template's later re-render can appear duplicated or missing. See [common-traps.md → LightDomSlotMixin composition with dynamic Lit children](https://github.com/barbaradenney/civui/blob/main/.claude/rules/common-traps.md) for the safe pattern. Static authoring (or wrapping the dynamic list in a stable container) works correctly.

:::

## Accessibility

- The container renders as `<ol role="list">` so screen readers announce sequence and count.
- Each item carries `role="listitem"` on its host element.
- The decorative rail (dot + connecting line) is wrapped in `aria-hidden="true"` — screen readers hear only the meaningful content (timestamp, actor, action, body).
- Timestamps are wrapped in `<time datetime="...">` when parseable, so assistive tech and indexers see a machine-readable ISO value. Unparseable values render as plain `<span>` text (HTML5 forbids invalid `datetime` attributes).

## Timestamp formats

| `timestamp-format` | Renders as |
|--------------------|-----------|
| `relative` | `3 hours ago` |
| `absolute` | `Jan 15, 2026, 10:30 AM` (locale-formatted) |
| `both` (default) | `Jan 15, 2026, 10:30 AM (3 hours ago)` |

The relative phrase is computed at render time with `Intl.RelativeTimeFormat` and uses the user's locale. The absolute label uses `Date.prototype.toLocaleString`.

## Intents

| `intent` | Dot color | Default icon |
|----------|-----------|--------------|
| `success` | success | `check` |
| `info` | info | `info` |
| `warning` | warning | `warning` |
| `error` | error | `error` |
| `neutral` _(default)_ | base | _(no icon — plain dot)_ |

Pass `icon=""` (empty string) on any item to suppress the default icon and show a plain colored dot. Pass `icon="some-name"` to use any icon from the [icon library](../core/icon).

## Usage

```html
<civ-timeline>
  <civ-timeline-item
    timestamp="2026-01-15T10:30:00Z"
    actor="Sarah Chen"
    action="Application submitted"
    intent="info">
  </civ-timeline-item>

  <civ-timeline-item
    timestamp="2026-01-16T14:20:00Z"
    actor="Benefits processing"
    action="Documentation requested"
    intent="warning">
    We need a copy of your DD-214 to verify service dates.
  </civ-timeline-item>

  <civ-timeline-item
    timestamp="2026-01-20T09:00:00Z"
    actor="J. Martinez (Reviewer)"
    action="Approved"
    intent="success">
    All documentation verified.
  </civ-timeline-item>
</civ-timeline>
```
