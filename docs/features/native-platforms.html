One Component, Every Platform

Schema-driven code generation for Web, iOS, and Android. Define a component once, generate native implementations from a single source of truth.

Schema-Driven Architecture

CivUI takes a schema-first approach to cross-platform components. Instead of writing the same component three times, you define it once as a platform-neutral schema and let the codegen pipeline produce idiomatic implementations for Lit (web), SwiftUI (iOS), and Jetpack Compose (Android).

Each schema is a declarative JSON document that fully describes a component. When the schema changes, every platform regenerates in lockstep — eliminating drift, reducing duplication, and guaranteeing that accessibility contracts, prop APIs, and event semantics remain identical everywhere.

What schemas capture

  • Props — typed properties with defaults, constraints, and documentation
  • Events — event names, detail shapes, and dispatch timing (input vs. change)
  • Accessibility contracts — required labels, ARIA roles, live region behavior, focus management
  • Render order — label, hint, error, control, supplementary info
  • Platform-specific mappings — keyboard types, visual transformations, native API equivalents

Side-by-Side Code

The same text input component across all three platforms. Each implementation is generated from a single text-input.schema.json definition and rendered idiomatically for its target.

WEB (Lit)
<civ-text-input
  label="Email address"
  name="email"
  type="email"
  hint="We'll use this to contact you"
  required
></civ-text-input>
iOS (SwiftUI)
CivTextInput(
    label: "Email address",
    value: $email,
    hint: "We'll use this to contact you",
    isRequired: true,
    inputType: .email
)
ANDROID (Jetpack Compose)
CivTextInput(
    label = "Email address",
    value = email,
    onValueChange = { email = it },
    hint = "We'll use this to contact you",
    isRequired = true,
    inputType = CivInputType.Email
)

Shared Design Tokens

Design tokens are defined once in W3C DTCG format and generated into CSS custom properties, Swift constants, and Kotlin values. Every platform gets the exact same visual language from the same source.

Accessibility Parity

Accessibility is not an afterthought on any platform. The schema encodes accessibility requirements, and each codegen target maps them to the platform's native accessibility APIs.

Package Structure

@civui/schema

Platform-neutral component definitions. The single source of truth for props, events, accessibility contracts, and validation across all platforms.

@civui/codegen

Code generator with Lit, SwiftUI, and Compose targets. Reads schemas and produces idiomatic, ready-to-use components.

@civui/forms

Web components built with Lit 3 and Light DOM. Text input, select, checkbox, radio, combobox, date picker, file upload, and more.

packages/ios/

Swift Package for iOS 16+ and macOS 13+. Native SwiftUI views with VoiceOver, Dynamic Type, and dark mode support.

packages/android/

Gradle module for API 26+ with Jetpack Compose. Material 3 foundation with CivUI design tokens and TalkBack semantics.

How It Works

The codegen CLI generates platform implementations from schemas. Run all platforms at once or target a single platform during development. 

pnpm generate              # All platforms
pnpm generate:ios          # iOS only
pnpm generate:android      # Android only
pnpm generate:web          # Web only
