1. Overview

The CivUI MCP Server is a Model Context Protocol server that connects AI coding assistants (Claude Desktop, Claude Code, Cursor, Windsurf, and other MCP-compatible tools) to the full CivUI design system. It provides structured tools for parsing, generating, validating, and testing government forms.

Instead of the AI assistant needing to learn CivUI component APIs from documentation, the MCP server gives it purpose-built tools that produce correct, accessible output every time. The server handles Section 508 compliance, WCAG 2.1 AA conformance, plain language requirements, and CivUI component patterns automatically.

2. Setup

Claude Desktop

Add this to your claude_desktop_config.json:

{
  "mcpServers": {
    "civui": {
      "command": "npx",
      "args": ["@civui/mcp-server"]
    }
  }
}

Claude Code

Add the server in your project settings or run:

claude mcp add civui npx @civui/mcp-server

Tier Selection

Control which tools are registered with the CIV_MCP_TIER environment variable:

{
  "mcpServers": {
    "civui": {
      "command": "npx",
      "args": ["@civui/mcp-server"],
      "env": {
        "CIV_MCP_TIER": "standard"
      }
    }
  }
}

Values: essential (13 tools), standard (~35 tools), all (83 tools, default).

3. The 3-Tier System

Tools are organized into three tiers to manage context window usage and keep AI assistants focused on the most relevant capabilities.

Tier Count When Registered Description
Essential 13 Always Core tools an LLM primarily needs: parse, generate, validate, gov forms, tests, style lookup. Covers 90% of use cases.
Advanced ~22 Standard + All modes Feature generators (signature, eligibility, save/resume), workflow tools, content tools, extra validation. Requested on demand.
Internal ~48 All mode only Composed by other tools. Repeatable sections, progress bars, print CSS, API handlers, bilingual forms, audit trails, etc. Usually called internally, not directly.

Recommendation: Start with standard tier for most projects. Use essential if context window is tight. Use all for full control over every generation step.

4. Quick Start Workflows

VA Form Generation

Know the form number? Two steps to a complete app.

  1. list_gov_forms
    See all 20 available forms with numbers, titles, and chapter counts
  2. assemble_gov_form(formNumber, format)
    Generate a complete working HTML or React app with routing, navigation, persistence, task list, and submission handling

Optional: validate_gov_form(formNumber) to check for issues.

PDF Form → Working App

Convert a legacy government PDF into accessible CivUI.

  1. parse_pdf_form(pdfBase64)
    Extract fields, types, labels, options, required flags → FormSchema
  2. generate_civui_form(schema)
    Convert FormSchema to CivUI web component HTML
  3. validate_form(html)
    Check for 508 violations (46 rules)
  4. generate_tests(html)
    Generate Vitest test suite for the form

HTML Form → CivUI

Modernize an existing HTML form.

  1. parse_html_form(html)
    Parse existing HTML into FormSchema
  2. generate_civui_form(schema)
    Generate accessible CivUI markup
  3. validate_form(html)
    Validate accessibility and compliance

Alt: generate_react_form(schema) for React TSX output.

Build from Scratch

Start from a template or your own schema.

  1. scaffold_from_template(name)
    Templates: Contact Form, Benefits Application, Change of Address, Authorization
  2. generate_civui_form(schema)
    Build from the template or your own FormSchema
  3. validate_form(html) + generate_tests(html)
    Check compliance and generate tests

How Form Definitions Work

Each form in the registry is a structured definition containing:

  • Chapters — logical sections of the form (personal info, contact, service history, etc.)
  • Fields — individual form fields with types, labels, validation rules, and conditional logic
  • Preparation items — list of documents and information the applicant needs before starting
  • Next steps — what happens after submission (processing time, follow-up actions)
  • Dynamic chapters — chapters that appear/disappear based on form answers
  • Cross-field rules — validation dependencies between fields

Generated output: The assemble_gov_form tool produces a single working HTML file with hash-based routing, task list navigation, chapter/page flow, form persistence (localStorage), progress tracking, error summary, review page with signature, and confirmation page. Use format: "react" for a multi-file React application.

9. Form Architecture

Every form generated by the MCP server follows a four-level hierarchy. This structure scales from a simple 3-field intent-to-file form to a complex 30-field disability compensation application.

The Form Hierarchy

Form Application ├── Introduction Page │ └── Purpose, eligibility, preparation checklist │ ├── Task List Hub (central navigation) │ ├── Progress bar (overall completion) │ └── Chapter groups with status tags │ ├── Chapters (logical sections) │ ├── Chapter 1: Personal Information │ │ ├── Page 1: Your name │ │ ├── Page 2: Date of birth │ │ └── Page 3: Social Security number │ │ │ ├── Chapter 2: Contact Information │ │ ├── Page 1: Mailing address │ │ ├── Page 2: Phone number │ │ └── Page 3: Email address │ │ │ ├── Chapter 3: Service History │ │ └── Page 1: Branch, dates, discharge (repeatable) │ │ │ └── Chapter 4: Supporting Documents │ └── Page 1: File upload │ ├── Review Page │ └── Summary of all answers + signature │ └── Confirmation Page └── Receipt, next steps, print

Chapters group related information (personal, contact, service). Users complete them in any order via the task list hub. Pages break chapters into focused steps. Each page asks one question or a small related group. The hub is always accessible — users can leave a chapter, come back later, and see their progress at a glance.

One Thing Per Page

CivUI follows the "one thing per page" pattern used by the UK Government Digital Service (GDS) and VA.gov. Each page asks one question or a small group of tightly related fields.

  • Reduces cognitive load — users focus on one decision at a time
  • Improves completion rates — 3 simple pages feels easier than 1 page with 15 fields
  • Better error handling — errors are caught immediately, not after scrolling
  • Mobile-friendly — no horizontal scrolling, no tiny touch targets
  • Screen reader friendly — clear focus management, one context at a time

What counts as "one thing": a single field ("What is your date of birth?"), a tightly related group (first/middle/last/suffix), an address block, or a yes/no question that may reveal follow-up fields.

Page Anatomy

┌─────────────────────────────────────────────┐ │ ‹ Back to task list (back link) │ │ │ │ APPLY FOR DISABILITY COMPENSATION │ │ Personal information (page header) │ │ VA Form 21-526EZ │ │ │ │ ● 1 ─── ○ 2 ─── ○ 3 (progress steps) │ │ Step 1 of 3 │ │ │ │ ┌─────────────────────────────────────┐ │ │ │ First name (required) │ │ │ │ ┌─────────────────────────────┐ │ │ │ │ │ │ │ │ │ │ └─────────────────────────────┘ │ │ │ └─────────────────────────────────────┘ │ │ │ │ [ Continue ] (action btn) │ └─────────────────────────────────────────────┘

Components used per page:

  1. <civ-link variant="back"> — Always at the top. Returns to the task list hub.
  2. <civ-page-header> — Eyebrow (form title), heading (chapter name), subheading (form number).
  3. <civ-progress-steps> — Shows each field in the chapter as a numbered step. Current step highlighted.
  4. Form field — One CivUI form component per page.
  5. <civ-button> — "Continue" advances; "Save and continue" completes the chapter.

Navigation and Progress

Task List Hub

The hub is the central navigation point. It shows all chapters as a task list with status tags:

  • Not started (blue) — Chapter has not been opened
  • In progress (teal) — Some fields completed
  • Complete (green) — All fields answered
  • Cannot start yet (gray) — Blocked (e.g., review requires all chapters complete)

A progress bar at the top shows overall completion: "2 of 5 sections complete" with a percentage.

Within a Chapter

Each chapter uses progress steps to show which field the user is on. Steps show as numbered circles — completed steps are filled, the current step is highlighted, upcoming steps are gray. Navigation: "Continue" advances, "Back" returns to previous step (from step 2 onward), "Back to task list" is always available at the top.

Dynamic and Conditional Forms

Conditional Fields

A field appears only when a previous answer triggers it. Uses the visibleWhen property in the FormSchema:

{
  "name": "insuranceProvider",
  "label": "Insurance provider name",
  "type": "text",
  "visibleWhen": {
    "field": "hasInsurance",
    "operator": "eq",
    "value": "yes"
  }
}

When a conditional field is hidden, it is removed from the step count. Progress steps update dynamically — if answering "No" hides 2 follow-up fields, the steps collapse from 5 to 3.

Dynamic Chapters

Entire chapters can appear or disappear based on form answers. The task list hub updates in real time — hidden chapters don't appear, and the progress bar adjusts its total.

Cross-Field Validation

Validations that depend on multiple fields ("Phone is required if contact method is call", "End date must be after start date"). Defined as cross-field rules in the schema or generated from plain English via generate_cross_field_rules.

Repeatable Sections

Some chapters collect the same set of fields multiple times — dependents, medical providers, service periods. CivUI handles this with the repeater pattern:

Chapter: Dependents ┌─────────────────────────────────────────────┐ │ Dependent 1 │ │ ┌─ Name: Jane Doe ────────────────────┐ │ │ │ Relationship: Spouse │ │ │ │ DOB: 01/15/1992 │ │ │ │ [Edit] [Remove] │ │ │ └─────────────────────────────────────┘ │ │ │ │ Dependent 2 │ │ ┌─ Name: John Doe Jr. ────────────────┐ │ │ │ Relationship: Child │ │ │ │ DOB: 06/22/2018 │ │ │ │ [Edit] [Remove] │ │ │ └─────────────────────────────────────┘ │ │ │ │ [ + Add another dependent ] │ │ │ │ [ Save and continue ] │ └─────────────────────────────────────────────┘

Key behaviors: completed items show as summary cards with edit/remove actions, "Add another" button respects max limits, field names auto-re-index on removal, and screen readers announce additions and removals.

Review and Submit

The review page shows a read-only summary of all answers, grouped by chapter. Each chapter has an "Edit" link that returns to that chapter. Sensitive fields (SSN, bank account) are masked in the summary.

Review your application ├── Personal information [Edit] │ Name: John Smith │ Date of birth: January 15, 1985 │ SSN: ***-**-1234 │ ├── Contact information [Edit] │ Address: 123 Main St, Washington, DC 20001 │ Phone: (555) 123-4567 │ Email: john@example.com │ └── Statement of truth ☒ I certify that the information I have provided is true and correct... Signature: John Smith Date: April 19, 2026 [ Submit application ]

After submission, the confirmation page shows: success alert with receipt, summary of what was submitted, next steps with expected timeline, print button, and help contact information.

Complex Workflows

Some forms involve multiple actors, approval processes, and post-submission amendments.

Multi-Actor Forms

Forms like the 21-22 (Power of Attorney) involve three roles: Veteran, Representative, and VA Reviewer. The workflow defines states (draft, submitted, under review, needs changes, approved, denied), transitions (who can move between states), and permissions (which sections each actor can edit in each state).

Workflow: Form 21-22 (Power of Attorney) ┌─────────┐ ┌───────────┐ ┌──────────────┐ │ Draft │────▶│ Submitted │────▶│ Under Review │ └─────────┘ └───────────┘ └──────┬───────┘ Veteran/Rep Veteran/Rep │ Reviewer ┌──────┴───────┐ ┌─────▼─────┐ ┌─────▼─────┐ │ Needs │ │ Approved │ │ Changes │ └───────────┘ └─────┬─────┘ ┌───────────┐ │ │ Denied │ ▼ └───────────┘ Resubmit → Submitted

Delegation

When a representative files on behalf of a Veteran: representative information section, legal attestation and authorization, consent signature from both parties. Generated by generate_delegation_sections.

Feedback and Review Cycles

Reviewers can leave inline comments on specific sections. Applicants see comments and can make changes before resubmitting. Comments track resolution status. Generated by generate_feedback_ui.

Post-Submission

  • Amendment flow — request changes after submission with diff (generate_amendment_flow)
  • Decision notices — formal approval/denial letters (generate_decision_notice)
  • Audit trail — timeline of all actions, state transitions, and comments (generate_audit_trail)
  • Case dashboard — combined workflow + progress + history view (generate_case_dashboard)

10. FormSchema Reference

FormSchema is the universal intermediate format that connects all tools. Every parse tool outputs a FormSchema, and every generation tool accepts one as input.

Minimal Example

{
  "title": "Change of Address",
  "action": "/api/submit",
  "method": "POST",
  "sections": [
    {
      "heading": "New Address",
      "fields": [
        { "name": "street", "label": "Street address", "type": "text", "required": true },
        { "name": "city", "label": "City", "type": "text", "required": true },
        { "name": "state", "label": "State", "type": "select", "required": true,
          "options": [
            { "label": "California", "value": "CA" },
            { "label": "Texas", "value": "TX" }
          ]
        },
        { "name": "zip", "label": "ZIP code", "type": "zip", "required": true }
      ]
    }
  ]
}

text, email, tel, url, number, password<civ-text-input>Set type accordingly
ssn<civ-text-input mask="ssn">Auto-masked, PII-flagged
zip<civ-text-input mask="zip">Auto-masked

select<civ-select>Requires options
combobox<civ-combobox>Typeahead search
radio<civ-radio-group>Requires options

checkbox-group<civ-checkbox-group>Multiple selections

date<civ-date-picker>For scheduling dates
file<civ-file-upload>Document upload
toggle<civ-toggle>On/off switch

Conditional Fields

{
  "name": "otherReason",
  "label": "Please describe",
  "type": "textarea",
  "visibleWhen": { "field": "reason", "operator": "eq", "value": "other" }
}

Cross-Field Validation

{
  "crossFieldRules": [
    {
      "id": "phone-required-if-call",
      "description": "Phone is required when contact method is call",
      "condition": { "field": "contactMethod", "operator": "eq", "value": "call" },
      "requiredFields": ["phone"]
    }
  ]
}

Or use generate_cross_field_rules with plain English: "Phone is required if contact method is call", "End date must be after start date", "Confirm email must match email address".
