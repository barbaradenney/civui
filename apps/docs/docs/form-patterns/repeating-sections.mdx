# Repeating Sections

Collect variable-length lists of items — dependents, service periods, previous addresses, employment history. `civ-repeater` has three modes; pick based on how much information you collect per item and whether each item deserves its own URL.

## Inline mode (1–3 fields per item)

Default. Every row's fields are visible on the page at the same time with add / remove buttons in line.

<StoryEmbed id="forms-form-repeater--default" />

```html
<civ-repeater legend="Dependents" name="dependents" item-label="dependent">
  <civ-text-input label="Full name" name="fullName"></civ-text-input>
</civ-repeater>
```

A more substantial inline example with multiple fields per row:

<StoryEmbed id="forms-form-repeater--multiple-fields" />

```html
<civ-repeater legend="Dependents" name="dependents" item-label="dependent">
  <civ-text-input label="First name" name="firstName"></civ-text-input>
  <civ-text-input label="Last name" name="lastName"></civ-text-input>
</civ-repeater>
```

## Form-steps mode (4+ fields, or multi-step entry)

Set `mode="form-steps"`. Existing rows render as compact summary cards; clicking **Add** or **Edit** opens a step-by-step entry flow, one section at a time. Use this when each item has enough fields that an inline row would overwhelm the page.

<StoryEmbed id="forms-form-repeater--form-steps-mode" />

```html
<civ-repeater
  legend="Dependents"
  name="dependents"
  item-label="dependent"
  mode="form-steps"
>
  <div data-step-label="Name">
    <civ-text-input label="First name" name="firstName" required></civ-text-input>
    <civ-text-input label="Last name" name="lastName" required></civ-text-input>
  </div>
  <div data-step-label="Contact">
    <civ-text-input label="Email" name="email" type="email"></civ-text-input>
    <civ-text-input label="Phone" name="phone" type="tel"></civ-text-input>
  </div>
</civ-repeater>
```

Form-steps mode also composes with the compound components — here, a dependents list that captures a name in step 1 and a relationship in step 2:

## Route mode (separate page per item)

Set `mode="route"`. The repeater is **controlled** — the host owns the `rows` array — and **Add** / **Edit** are real `<a href>` links pointing at host-rendered pages. Browser back, deep linking, "open in new tab", and save-and-resume all work naturally. Use this when each item is complex enough that filling it deserves its own page (8+ fields, multi-section items, items the user might come back to tomorrow).

<StoryEmbed id="forms-form-repeater--route-mode" />

```html
<civ-repeater
  legend="Your dependents"
  item-label="dependent"
  mode="route"
  add-href="/dependents/new"
  edit-href-pattern="/dependents/{id}/edit"
  id-field="id"
  summary-fields="firstName,lastName"
  .rows=${this.dependents}
  @civ-repeater-remove=${(e) => this.removeDependent(e.detail.id)}
></civ-repeater>
```

Slotted children are ignored in this mode — the form lives on the destination page that the host renders. **Remove** fires `civ-repeater-remove` with `{ index, id, row }`; the host updates its own array and re-passes via `.rows`.

## Min and max constraints

Set `min` and `max` to control the allowed number of items:

<StoryEmbed id="forms-form-repeater--min-max" />

```html
<!-- At least 1 service period, up to 10 -->
<civ-repeater legend="Service periods" name="service" item-label="service period" min="1" max="10">
  ...
</civ-repeater>
```

## Which mode to use

| Fields per item | URL changes? | Mode | Example |
|----------------|--------------|------|---------|
| 1–3 simple fields | No | `inline` (default) | Phone numbers, email addresses |
| 4–10 fields, stay on page | No | `form-steps` | Inline wizard for dependents, employment history |
| 8+ fields, own page / save-and-resume | Yes | `route` | Dependent profile (name + address + SSN + employment) on its own URL |

## Confirming destructive removals

The default Remove button is one-click — fine for short scratch lists. When removed content is harder to recover (typed dependent info, uploaded files), add a confirmation step. CivUI's rule is modal-only — see [Destructive actions](/foundations/destructive-actions) for the full pattern and the [repeater docs](/components/form/repeater#adding-confirmation-to-remove) for the wiring snippet.
