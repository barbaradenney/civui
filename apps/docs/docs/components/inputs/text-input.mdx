`<civ-text-input>` `<civ-zip>`

Accessible text input with label, hint, error, and width variants. Supports input masking, declarative validation, prefix/suffix text, and inline icons. Uses ElementInternals for native form participation. The remaining single-input ZIP preset is documented [below](#specialized-presets); contact fields live on [**Contact**](./contact.mdx), sensitive identifiers on [**PII**](./pii.mdx), and numeric inputs (dollar amounts, routing numbers, generic numbers) on [**Number**](./number.mdx).

### Character Counter

When `maxlength` is set, the input renders a "characters remaining" counter below the field. Visual count updates immediately; screen-reader announcements debounce on a 1s interval to avoid spamming. The counter is suppressed when a mask is active (the mask drives the length constraint).

### Clearable

Set `clearable` to render an inline "×" button on the trailing edge whenever the value is non-empty. Clicking it clears the value, refocuses the input, and fires `civ-input` + `civ-change` with `{ value: "" }`. The clear button takes precedence over any `trailing-icon` while the field has content.

<StoryEmbed id="forms-inputs-text-input--clearable" />

### Trailing Action Slot

For affordances CivUI doesn't ship as props — copy-to-clipboard, paste, scan, generate, units toggle — pass a child element with the `data-trailing-action` attribute. Apply the `.civ-input-action` class so the button matches the input chrome (height, hover state, focus ring). Set `type="button"` and an `aria-label`. The slot is **web-only** (Light DOM design system; iOS/Android/Drupal do not project this) and is intentionally an escape hatch — if a use case becomes common, we'll promote it to a first-class prop.

```html
<civ-text-input label="API key" name="apiKey" value="sk-...">
  <button data-trailing-action type="button" class="civ-input-action" aria-label="Copy API key">
    <civ-icon name="content_copy" aria-hidden="true"></civ-icon>
  </button>
</civ-text-input>
```

<StoryEmbed id="forms-inputs-text-input--trailing-action-slot" />

#### Trailing region order

The trailing inset region inside the input holds at most one component-rendered button at a time, plus any consumer-provided slot content. **Component-rendered buttons follow this precedence** (highest wins):

1. **Clear button** — when `clearable` is set and the value is non-empty
2. **Reveal-password button** — when `reveal-password` is set and `type="password"`
3. **Trailing-action slot** — consumer-provided children with `data-trailing-action`
4. **Trailing icon** — passive icon from the `trailing-icon` prop
5. **Suffix** — text label (mutually exclusive with all of the above; `suffix` always renders when set)

Two pragmatic consequences:
- **Only one component-rendered button shows at a time.** Setting `clearable` + `reveal-password` together is fine — clear shows while the field has a value, reveal takes over when the field is empty.
- **Slot content renders in source order.** If you pass multiple children with `data-trailing-action`, they appear left-to-right in DOM order, immediately to the left of any component-rendered button. There's no "slot 1" / "slot 2" API — just put your buttons in the order you want.

The inline-action pattern (text-style buttons inset in the input chrome — `.civ-input-inline-action`) is exclusive to `civ-text-input` and to component-managed buttons inside `civ-date-picker` (the "Today" shortcut). **Combobox-shaped fields** (`civ-combobox`, `civ-country`, `civ-time-picker` combo mode) already have a clear button plus a chevron/dropdown trigger and can't fit additional inline buttons without crowding. Reach for the footer-button pattern (`show-now-button` on time-picker, or a sibling `civ-button`) on those surfaces instead.

## Custom Mask Patterns

For formats not covered by the specialized components, use `mask-pattern` with slot syntax:

| Slot | Matches |
|------|---------|
| `#` | Digit (0-9) |
| `A` | Letter (a-z, A-Z) |
| `*` | Any printable character |

All other characters are inserted as literals.

### Mask Modes

- **`blur`** (default): User types freely. Mask applied on blur for display. Accessible and predictable.
- **`live`**: Formats as user types. Has accessibility tradeoffs (cursor jumping).

## Specialized Presets

These wrappers extend `<civ-text-input>` with pre-configured masking, validation, default labels, hints, and HTML attributes. They share the same events (`civ-input`, `civ-change` with `{ value: string }` detail). The presets below are general-purpose text-input wrappers; identifier presets that need PII protection are on the [**PII**](./pii.mdx) page, contact presets on [**Contact**](./contact.mdx), and numeric presets (dollar amounts, routing numbers) on [**Number**](./number.mdx).

### ZIP Code (`<civ-zip>`)

Pre-configured input for US ZIP codes with automatic masking and validation. Supports standard 5-digit and extended ZIP+4 formats. Used as a leaf field in `<civ-address>`.

Default label: "ZIP code"

| Prop | Type | Default | Description |
|------|------|---------|-------------|
| `extended` | `boolean` | `false` | Enable ZIP+4 format (e.g., 12345-6789) |

<StoryEmbed id="forms-inputs-text-input--zip-code" />

<StoryEmbed id="forms-inputs-text-input--zip-plus-4" />

```html
<!-- Standard 5-digit ZIP code -->
<civ-zip label="ZIP code" name="zip" required></civ-zip>

<!-- ZIP+4 format -->
<civ-zip label="ZIP code" name="zip" extended required></civ-zip>
```

## Usage

```html
<!-- Basic text input -->
<civ-text-input label="Full name" name="fullName" required></civ-text-input>

<!-- Custom mask pattern -->
<civ-text-input label="Case number" hint="For example: ABC-1234" name="caseNumber" mask-pattern="AAA-####"></civ-text-input>

<!-- With prefix -->
<civ-text-input label="Social handle" name="handle" prefix="@" width="md"></civ-text-input>

<!-- With suffix -->
<civ-text-input label="Weight" name="weight" suffix="lbs" inputmode="decimal" width="sm"></civ-text-input>

<!-- With character counter -->
<civ-text-input label="Short bio" name="bio" maxlength="60"></civ-text-input>
```
