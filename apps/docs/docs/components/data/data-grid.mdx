`<civ-data-grid>`

Semantic `<table>`-based data grid for admin and back-office screens. Renders sortable column headers, selectable rows, and per-row action menus. Compose with a sibling `<civ-pagination>` for paged data. On mobile (≤480px), rows collapse to vertical label/value blocks via CSS — no JavaScript re-template.

The grid is **controlled** — the consumer manages `rows`, `columns`, `sortBy`, `sortDirection`, `page`, and `pageSize`. The grid dispatches `civ-sort`, `civ-selection-change`, and `civ-row-action`; the consumer listens for `civ-page-change` directly on the pagination element.

## Accessibility foundation: semantic `<table>`, not `role="grid"`

The WAI-ARIA Authoring Practices Guide recommends semantic `<table>` over `role="grid"` for predominantly readable tabular data — even when columns are sortable. `role="grid"` requires full 2D arrow-key navigation that's worse for screen reader users on data display. CivUI's data grid follows this guidance:

- Native `<table>` + `<caption>` for the accessible name
- `<th scope="col">` for column headers
- `aria-sort="ascending|descending|none"` on sortable headers
- Real `<button>` inside each sortable `<th>` (the focusable affordance)
- Row-action menus use `<civ-menu>` (WAI-ARIA Menu Button pattern)
- Empty / loading / error states use `aria-live` / `role="alert"` so screen readers hear them

If you need spreadsheet-style 2D arrow-key navigation, the right answer is a separate component layered on `role="grid"` — don't promote the data grid in place.

## Column definition

The `columns` JS property accepts an array of `GridColumn` objects:

```ts
interface GridColumn {
  /** Stable identifier — keys into row.cells and identifies the column in sort events. */
  key: string;
  /** Header text rendered in <th>. */
  header: string;
  /** Becomes a sort button and dispatches civ-sort on click. */
  sortable?: boolean;
  /** Horizontal alignment. Use 'numeric' to enable tabular-nums for number columns. */
  align?: 'start' | 'center' | 'end' | 'numeric';
  /** CSS width (e.g. '12rem', '25%'). */
  width?: string;
  /** Optional formatter — return a string, number, or Lit TemplateResult for badges, links, etc. */
  formatter?: (value: unknown, row: GridRow, rowIndex: number) => string | number | TemplateResult;
}
```

## Row definition

```ts
interface GridRow {
  /** Stable identifier — required for selection and row-action correlation. */
  id: string;
  /** Cell values keyed by column.key. The raw value passed to a column's formatter. */
  cells: Record<string, unknown>;
  /** When true, the row is non-interactive. */
  disabled?: boolean;
  /** Row-action menu entries. Omit or pass an empty array to suppress the actions column. */
  actions?: GridRowAction[];
}

interface GridRowAction {
  id: string;
  label: string;
  destructive?: boolean;  // red styling for delete-style actions
  disabled?: boolean;
  href?: string;          // renders as <a> instead of <button>
}
```

## Master-detail (interactive rows)

Set `interactive` on the grid to make rows clickable. Clicks fire a
`civ-row-activate` event with the row's data — wire it to a sibling
`civ-drawer` (or any other detail surface) to build a master-detail flow:

```ts
const grid = document.querySelector('civ-data-grid')!;
const drawer = document.querySelector('civ-drawer')!;

grid.interactive = true;
grid.addEventListener('civ-row-activate', (e) => {
  const { row } = e.detail;
  drawer.heading = `Application ${row.id}`;
  renderDetail(row);  // your code: populate the drawer's slotted content
  drawer.open = true;
});
```

**What's ignored.** The row-click handler short-circuits when the click
target is inside the selection checkbox cell, the row-actions cell, or
any interactive descendant (`<a>`, `<button>`, `<input>`, `<select>`,
`<textarea>`, `[role="button"]`). Those affordances retain their own
behavior — toggling a checkbox doesn't also open the detail, and clicking
the kebab menu doesn't activate the row.

**Disabled rows** (`row.disabled: true`) skip activation entirely — they
get neither the interactive class nor the click handler.

**Selection and activation are independent.** When `selectable` is set,
clicking the row body fires `civ-row-activate` but does NOT toggle the
row's selection. Selection lives on the checkbox in the select cell.

**Keyboard** is intentionally not supported at the row level in v1. The
WAI-ARIA Authoring Practices guidance for `<table>`-based grids
discourages overriding `<tr>` with `role="button"` since it confuses the
row semantic. The component **warns in dev mode** when `interactive=true`
has no row actions, because the result is a mouse-only affordance. Wire
a "View details" action that fires the same destination so keyboard /
switch-control users have a focusable equivalent:

```ts
function toRows(records): GridRow[] {
  return records.map((r) => ({
    id: r.id,
    cells: { … },
    actions: [
      // The keyboard equivalent of clicking the row body. Both paths
      // dispatch through the consumer's openDetail() routine.
      { id: 'view', label: `View details for ${r.id}` },
      // … other actions
    ],
  }));
}

grid.addEventListener('civ-row-activate', (e) => openDetail(e.detail.row));
grid.addEventListener('civ-row-action', (e) => {
  if (e.detail.action === 'view') openDetail(e.detail.row);
});
```

## Responsive behavior

Two modes via the `responsive` prop:

- **`stacked` (default)** — On ≤480px, each row collapses into a vertical block. Each `<td>` renders its column header (from `data-label`) inline next to the value. The USWDS stacked table pattern.
- **`scroll`** — Wraps the table in a horizontally-scrollable region. Rows stay tabular at all viewport widths. Use when columns must stay in row order on small screens.

Both modes preserve the same DOM — CSS does the work, so screen readers always see a proper `<table>`.

## Server-side data

The grid is controlled, so server-side sorting and pagination are natural:

```ts
const grid = document.querySelector('civ-data-grid')!;
const pager = document.querySelector('civ-pagination')!;

let state = { page: 1, pageSize: 25, sortBy: '', sortDirection: 'none' as const };

async function fetchAndRender() {
  grid.loading = true;
  try {
    const { rows, total } = await api.search(state);
    grid.rows = rows;
    pager.totalItems = total;
    pager.page = state.page;
    grid.loading = false;
  } catch (err) {
    grid.errorMessage = err instanceof Error ? err.message : 'Failed to load';
    grid.loading = false;
  }
}

grid.addEventListener('civ-sort', (e) => {
  state.sortBy = e.detail.column;
  state.sortDirection = e.detail.direction;
  state.page = 1; // reset paging when sort changes
  fetchAndRender();
});

pager.addEventListener('civ-page-change', (e) => {
  state.page = e.detail.page;
  state.pageSize = e.detail.pageSize;
  fetchAndRender();
});
```
