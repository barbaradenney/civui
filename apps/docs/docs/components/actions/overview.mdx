# Actions

The Actions category covers every component that does or navigates something when the user activates it. CivUI ships **eleven** action-shaped affordances — a wider catalog than most design systems on purpose, because the right affordance is rarely the same as the next-loudest one.

If you're not sure which to reach for, start here.

## Pick the right one

| What the user is doing | Use |
|---|---|
| Submitting a form, confirming a destructive action, primary CTA on a page | [`<civ-button>`](./button) |
| Toolbar action, row action, kebab / overflow trigger, compact contextual action | [`<civ-action-button>`](./action-button) |
| Small click affordance in flowing content (helper rows, inline triggers, quiet shortcuts) | [`<civ-text-button>`](./text-button) |
| Copy / Paste / Scan / Generate — fire-and-forget with a "✓ Copied" receipt | [`<civ-confirm-button>`](./confirm-button) |
| Show / Hide, Mute / Unmute — two persistent labels and `aria-pressed` | [`<civ-toggle-button>`](./toggle-button) |
| Quick-filter row (Today / Last 7 days), suggestion chip, fire-and-forget chip | [`<civ-action-chip>`](./action-chip) |
| Toggleable filter selection (with check icon + optional remove) | [`<civ-filter-chip>`](./filter-chip) |
| User-entered token (recipient list, tags, applied-filter readout) | [`<civ-input-chip>`](./input-chip) |
| Inline link inside body prose (sentence-level navigation) | [`<civ-link>`](./link) |
| Card-shaped clickable surface that navigates | [`<civ-link-card>`](./link-card) |
| Accessibility skip-to-main-content link | [`<civ-skip-link>`](./skip-link) |

## All action components

### Buttons

`<civ-button>` — the page-level CTA. Heavy chrome, 44px tap-target floor, lives in form rows, button rows, page footers. Accepts `href` to render as a link with button chrome.

<StoryEmbed id="actions-button--default" />

[Open civ-button →](./button)

---

`<civ-action-button>` — compact toolbar / row-action button with the same emphasis system (`primary` / `secondary` / `tertiary`) at smaller padding. Use inside cards, list rows, table cells, toolbars. Supports a tri-state `pressed` for toggle affordances.

<StoryEmbed id="actions-action-button--default" />

[Open civ-action-button →](./action-button)

---

`<civ-text-button>` — the small text-button primitive. Same `emphasis` system as the others; no built-in behavior beyond a `civ-click` event. Reach for it when you want the text-btn visual idiom without the confirm-receipt or toggle-press machinery.

<StoryEmbed id="actions-text-button--default" />

[Open civ-text-button →](./text-button)

---

`<civ-confirm-button>` — text-button + a transient ✓ receipt. Wires `aria-live="polite"` during the success window so screen readers re-announce the receipt label. Consumer does the actual work in the `civ-confirm` listener.

<StoryEmbed id="actions-confirm-button--default" />

[Open civ-confirm-button →](./confirm-button)

---

`<civ-toggle-button>` — text-button + persistent two-state press. Renders `aria-pressed="true|false"` on the inner `<button>` (the standard ARIA toggle pattern). Reflects `pressed` to the host attribute for two-way binding.

<StoryEmbed id="actions-toggle-button--default" />

[Open civ-toggle-button →](./toggle-button)

### Chips

`<civ-action-chip>` — outlined rounded pill that fires `civ-click` and stays visually unchanged. No toggle, no check icon. Use for suggestion chips, quick-filter rows that immediately re-fetch.

<StoryEmbed id="actions-action-chip--default" />

[Open civ-action-chip →](./action-chip)

---

`<civ-filter-chip>` — toggleable selection chip with a check icon on the selected state. Pairs with `<civ-filter-chip-group>` for roving-tabindex keyboard nav across a row. For removable user-entered tokens, use `<civ-input-chip>` instead.

<StoryEmbed id="actions-filter-chip--default" />

[Open civ-filter-chip →](./filter-chip)

---

`<civ-input-chip>` — display chip representing user-entered content with an always-present remove handle. Use for recipient lists, applied-filter readouts, typed tags.

<StoryEmbed id="actions-input-chip--default" />

[Open civ-input-chip →](./input-chip)

`<civ-link>` — inline prose link. Always rendered underlined for accessibility. Use inside sentences and paragraph body text.

## Shared emphasis vocabulary

`civ-button`, `civ-action-button`, `civ-text-button`, `civ-confirm-button`, and `civ-toggle-button` all share the same `emphasis="primary | secondary | tertiary"` prop. The visual weight is the same axis across the family — pick the **component** by behavior, and the **emphasis** by how loud the affordance should read in its surface.

The chips (`civ-action-chip`, `civ-filter-chip`, `civ-input-chip`) are a different visual idiom (outlined rounded pills) and don't share the emphasis vocabulary directly — `civ-filter-chip` has its own `emphasis="primary | secondary"` controlling the *selected-state* fill weight, not the overall component prominence.

## Adjacent categories

- [Layout → button-group](../layout/button-group) — connected/flush toolbar of buttons (`role="toolbar"`).
- [Layout → disclosure](../layout/disclosure) and [layout → read-more](../layout/read-more) — text-btn-styled triggers for native `<details>` and prose truncation. They compose the same `.civ-text-btn` chrome but own their own behavior contracts.
- [Overlays → menu](../overlays/menu) — anchored kebab / overflow menus typically triggered by a `civ-action-button`.

## Anti-patterns

| ✗ Don't | ✓ Do instead |
|---|---|
| Use `<civ-button>` for an inline Copy affordance | Use `<civ-confirm-button>` in the input helper row |
| Use `<civ-button>` for a small toolbar trigger | Use `<civ-action-button>` |
| Reach for the deprecated `variant="chip"` / `variant="inline"` on confirm/toggle | Use the new `emphasis="secondary"` / `emphasis="tertiary"` |
| Hand-roll `<button class="civ-text-btn civ-text-btn--chip">` | Use `<civ-text-button>` — it's now a real element |
| Wrap two `<civ-button>` clusters in `<div class="civ-flex civ-gap-3">` | Use `<div class="civ-button-row">` so each button goes full-width on mobile |
| Use `<civ-filter-chip>` for a user-entered email tag | Use `<civ-input-chip>` (no selected state, always removable) |
| Render `<a class="civ-link">` directly inside body prose | Use `<civ-link>` — same `<a>` but the contract surface |
| Add `success-label` expectations to `<civ-button>` | The receipt-state machine lives on `<civ-confirm-button>`; for "Save → Saved ✓" today, swap the label manually after the async work resolves |
