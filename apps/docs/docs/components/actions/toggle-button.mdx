`<civ-toggle-button>`

Text button with two persistent labels and an `aria-pressed` state. Click flips `pressed` and swaps the visible label. Use for show/hide toggles where each state has a clear name — password reveal (Show / Hide), mute / unmute, expand / collapse on a custom surface.

The `pressed` property reflects to the host attribute, so consumers can two-way bind by setting `.pressed` externally or by listening for `civ-toggle`. Padding stays stable across the swap — the variant class is held constant; only the label text and `aria-pressed` flip.

## When to use vs. not

| Need | Use |
|---|---|
| Two persistent labels with a press state (Show / Hide, Mute / Unmute) | `<civ-toggle-button>` |
| Fire-and-forget action with receipt (Copy / Paste / Scan) | `<civ-confirm-button>` |
| Native disclosure (open/closed `<details>`) | `<civ-disclosure>` |
| Read-more expansion of prose | `<civ-read-more>` |
| Primary form CTA (Submit, Continue) | `<civ-button>` |

## Two-way binding

The `pressed` property reflects to the host attribute, so:

```ts
const btn = document.querySelector('civ-toggle-button')!;
// React to clicks
btn.addEventListener('civ-toggle', (e) => {
  console.log('now pressed:', e.detail.pressed);
});
// Or drive it externally
btn.pressed = true;
```

Setting `pressed` programmatically updates the rendered label and `aria-pressed` immediately — it doesn't fire `civ-toggle` (the event signifies a user-initiated transition, not a state sync).

## Accessibility

- The inner `<button>` carries `aria-pressed="true"` or `"false"` — never absent. This is the standard ARIA toggle-button pattern.
- Both labels are required; there's no icon-only mode because a glyph alone leaves AT users without a name for the pressed state.
- Disabled buttons set both `disabled` and ignore clicks (no `pressed` flip, no `civ-toggle` event).
