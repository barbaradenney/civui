# Confirm Button

<!-- contract-link:start -->

> **Contract reference:** [`civ-confirm-button`](pathname:///civui/storybook/?path=/docs/contract-civ-confirm-button--docs). Auto-generated structured contract for cross-platform implementation.

<!-- contract-link:end -->

`<civ-confirm-button>`

Text button that fires an action and shows a transient ✓ confirmation before reverting. Used for fire-and-forget actions whose feedback should be visible inline without yanking the user to a toast — Copy, Paste, Scan, Generate.

Shares the [text-button](./text-button) chrome and `emphasis` levels (`primary` / `secondary` / `tertiary`) — pick the emphasis by visual weight, pick the component by behavior. For a plain click without the receipt machinery, use [`civ-text-button`](./text-button); for two-state toggles use [`civ-toggle-button`](./toggle-button).

The consumer does the actual work in the `civ-confirm` event listener; the component manages the success-window timing and visual swap. **Padding stays stable across states** (the variant class is held constant; only `.is-success` is added), so the button doesn't visibly shrink mid-confirmation.

## When to use vs. not

| Need | Use |
|---|---|
| Fire-and-forget action with receipt (Copy, Paste, Scan, Generate) | `<civ-confirm-button>` |
| Two-state toggle (Show / Hide, mute / unmute) | `<civ-toggle-button>` |
| Plain text button with no built-in behavior | [`<civ-text-button>`](./text-button) |
| Primary form CTA (Submit, Continue) | `<civ-button>` |
| Toolbar action (Edit, Delete, …) | `<civ-action-button>` |

## Handling failure

`civ-confirm` fires synchronously the moment the user clicks, and the success window starts immediately. If the consumer's work (clipboard write, network request) might fail asynchronously, call `.reset()` from the rejection path to dismiss the receipt early:

```ts
const btn = document.querySelector('civ-confirm-button')!;
btn.addEventListener('civ-confirm', async () => {
  try {
    await someAsyncWork();
  } catch (err) {
    btn.reset();
    showError(err);
  }
});
```

## Accessibility

- The receipt label is announced via `@civui/core`'s shared polite live-region queue on click — a same-button text swap doesn't otherwise trigger a screen-reader re-read. The button itself does **not** carry `aria-live`; toggling it on the button at the same render commit as the text change is racy on NVDA/JAWS (the region must already be observed as live BEFORE its content mutates).
- The check-icon decoration is a CSS `::before` mask with no semantic role; the label text carries the meaning.
- The success color (`--civ-color-success-dark`) hits AA contrast on the surrounding chip background. Don't rely on color alone — the label text changes too.
