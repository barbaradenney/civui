# Data Grid

<!-- contract-link:start -->

> **Contract reference:** [`civ-data-grid`](pathname:///civui/storybook/?path=/docs/contract-civ-data-grid--docs). Auto-generated structured contract for cross-platform implementation.

<!-- contract-link:end -->

`<civ-data-grid>`

Semantic `<table>`-based data grid for admin and back-office screens. Renders sortable column headers, selectable rows, and per-row action menus. Compose with a sibling `<civ-pagination>` for paged data. On mobile (≤480px), rows collapse to vertical label/value blocks via CSS. No JavaScript re-template.

The grid is **controlled**. The consumer manages `rows`, `columns`, `sortBy`, `sortDirection`, `page`, and `pageSize`. The grid dispatches `civ-sort`, `civ-selection-change`, and `civ-row-action`; the consumer listens for `civ-page-change` directly on the pagination element.

## Accessibility foundation: semantic `<table>` by default, `role="grid"` on demand

The WAI-ARIA Authoring Practices Guide recommends semantic `<table>` over `role="grid"` for predominantly readable tabular data. Even when columns are sortable. CivUI's data grid follows that guidance **by default**:

- Native `<table>` + `<caption>` for the accessible name
- `<th scope="col">` for column headers
- `aria-sort="ascending|descending|none"` on sortable headers
- Real `<button>` inside each sortable `<th>` (the focusable affordance)
- Row-action menus use `<civ-menu>` (WAI-ARIA Menu Button pattern)
- Empty / loading / error states use `aria-live` / `role="alert"` so screen readers hear them

For admin screens with many interactive cells (sort, expand, edit, select, row actions) Tabbing through every control becomes noisy. Set the `keyboardNav` prop and the grid promotes to `role="grid"` with 2D arrow-key navigation per the [WAI-ARIA Grid Pattern](https://www.w3.org/WAI/ARIA/apg/patterns/grid/). The whole grid becomes one tab stop; arrow keys move between cells; Enter / Space / F2 activate the cell's primary control. See the [Keyboard navigation](#keyboard-navigation) section below for the full key map.

## Column definition

The `columns` JS property accepts an array of `GridColumn` objects:

```ts
interface GridColumn {
  /** Stable identifier — keys into row.cells and identifies the column in sort events. */
  key: string;
  /** Header text rendered in <th>. */
  header: string;
  /** Becomes a sort button and dispatches civ-sort on click. */
  sortable?: boolean;
  /** Horizontal alignment. Use 'numeric' to enable tabular-nums for number columns. */
  align?: 'start' | 'center' | 'end' | 'numeric';
  /** CSS width (e.g. '12rem', '25%'). */
  width?: string;
  /** Optional formatter — return a string, number, or Lit TemplateResult for badges, links, etc. */
  formatter?: (value: unknown, row: GridRow, rowIndex: number) => string | number | TemplateResult;
}
```

## Multi-column sort

By default, the grid is single-sort: clicking a header cycles `asc → desc → none` on that column alone. Set `multiSort` to opt into stacked sort keys. The Excel / Sheets behavior where Shift-click adds a secondary key, then a tertiary, etc.

```html
<civ-data-grid caption="Applications" multi-sort></civ-data-grid>
```

### State shape

When `multiSort` is on, `sortKeys` is the source of truth:

```ts
interface GridSortKey {
  key: string;
  direction: 'asc' | 'desc';
}

grid.sortKeys = [
  { key: 'status',  direction: 'asc' },   // primary
  { key: 'updated', direction: 'desc' },  // secondary
];
```

`sortBy` / `sortDirection` continue to reflect the **primary** (first) key. The grid mirrors `sortKeys[0]` into them on every render when `multiSort` is on, so backward-compat readers see consistent state. **Writes** to `sortBy` / `sortDirection` while `multiSort` is on are overwritten on the next render; manage `sortKeys` and let the mirror fan out.

### Interaction model

- **Plain click** on any header. Replaces the stack with a single key on that column, cycling `asc → desc → none` if the column is already in the stack (at any position).
- **Shift-click** on a header NOT in the stack. Appends it (asc).
- **Shift-click** on a header already in the stack. Cycles its direction `asc → desc → removed`.

Keyboard users get the same gestures: Enter / Space activate as a plain click, and **Shift+Enter / Shift+Space** acts as Shift-click. This works whether or not `keyboardNav` is on; with `keyboardNav` the focus stop is the cell itself, otherwise it's the inner sort button.

Position badges (`1, 2, 3…`) render next to the chevron when the stack has more than one key, so the priority order is visible. A single-key stack omits the badge (no "1" on a column that's the only sort). Screen reader users hear the position in the sort button's `aria-label`. E.g. "Sort Status descending, sort priority 2".

### Event detail

Every `civ-sort` event carries both shapes:

```ts
grid.addEventListener('civ-sort', (e) => {
  const { column, direction, sortKeys } = e.detail;
  // - column / direction describe the JUST-toggled column (single-sort consumers can use this)
  // - sortKeys is the full target stack (multi-sort consumers use this)
  grid.sortKeys = sortKeys;
  rerenderRowsWithSort(sortKeys);
});
```

For a client-side multi-key sort:

```ts
// Compare two values from the same column. Numeric compare for numbers
// (avoids the '10' < '2' lex trap), string compare for everything else.
function compare(a, b) {
  if (typeof a === 'number' && typeof b === 'number') return a - b;
  const as = String(a ?? '');
  const bs = String(b ?? '');
  return as < bs ? -1 : as > bs ? 1 : 0;
}

function applySort(rows, keys) {
  if (keys.length === 0) return rows;
  return [...rows].sort((a, b) => {
    for (const { key, direction } of keys) {
      const cmp = compare(a.cells[key], b.cells[key]);
      if (cmp !== 0) return direction === 'asc' ? cmp : -cmp;
    }
    return 0;
  });
}
```

For typed columns beyond plain numbers and strings (dates, booleans, currency strings like `"$1,250"`), supply per-column comparators of your own.

For server-side, hand the array to your query API. Most SQL engines accept it as an `ORDER BY` clause.

## Row definition

```ts
interface GridRow {
  /** Stable identifier — required for selection and row-action correlation. */
  id: string;
  /** Cell values keyed by column.key. The raw value passed to a column's formatter. */
  cells: Record<string, unknown>;
  /** When true, the row is non-interactive. */
  disabled?: boolean;
  /** Row-action menu entries. Omit or pass an empty array to suppress the actions column. */
  actions?: GridRowAction[];
}

interface GridRowAction {
  id: string;
  label: string;
  destructive?: boolean;  // red styling for delete-style actions
  disabled?: boolean;
  href?: string;          // renders as <a> instead of <button>
}
```

## Master-detail (interactive rows)

Set `interactive` on the grid to make rows clickable. Clicks fire a
`civ-row-activate` event with the row's data. Wire it to a sibling
`civ-drawer` (or any other detail surface) to build a master-detail flow:

```ts
const grid = document.querySelector('civ-data-grid')!;
const drawer = document.querySelector('civ-drawer')!;

grid.interactive = true;
grid.addEventListener('civ-row-activate', (e) => {
  const { row } = e.detail;
  drawer.heading = `Application ${row.id}`;
  renderDetail(row);  // your code: populate the drawer's slotted content
  drawer.open = true;
});
```

**What's ignored.** The row-click handler short-circuits when the click
target is inside the selection checkbox cell, the row-actions cell, or
any interactive descendant (`<a>`, `<button>`, `<input>`, `<select>`,
`<textarea>`, `[role="button"]`). Those affordances retain their own
behavior. Toggling a checkbox doesn't also open the detail, and clicking
the kebab menu doesn't activate the row.

**Disabled rows** (`row.disabled: true`) skip activation entirely. They
get neither the interactive class nor the click handler.

**Selection and activation are independent.** When `selectable` is set,
clicking the row body fires `civ-row-activate` but does NOT toggle the
row's selection. Selection lives on the checkbox in the select cell.

**Keyboard** is intentionally not supported at the row level in v1. The
WAI-ARIA Authoring Practices guidance for `<table>`-based grids
discourages overriding `<tr>` with `role="button"` since it confuses the
row semantic. The component **warns in dev mode** when `interactive=true`
has no row actions, because the result is a mouse-only affordance. Wire
a "View details" action that fires the same destination so keyboard /
switch-control users have a focusable equivalent:

```ts
function toRows(records): GridRow[] {
  return records.map((r) => ({
    id: r.id,
    cells: { … },
    actions: [
      // The keyboard equivalent of clicking the row body. Both paths
      // dispatch through the consumer's openDetail() routine.
      { id: 'view', label: `View details for ${r.id}` },
      // … other actions
    ],
  }));
}

grid.addEventListener('civ-row-activate', (e) => openDetail(e.detail.row));
grid.addEventListener('civ-row-action', (e) => {
  if (e.detail.action === 'view') openDetail(e.detail.row);
});
```

## Expandable rows

Set `row.expandable: true` on rows that should reveal an inline detail
panel. When ANY row in the data set is expandable, the grid renders a
leading chevron column. Non-expandable rows get an empty cell in that
column; expandable rows get a chevron button that toggles between
`chevron-right` and `chevron-down`.

The detail content comes from `grid.expandTemplate`. A function
`(row) => string | number | TemplateResult` you set as a JS property:

```ts
grid.rows = [
  { id: '1', cells: { name: 'Smith' }, expandable: true },
  { id: '2', cells: { name: 'Doe' } },        // not expandable; empty cell
  { id: '3', cells: { name: 'Reyes' }, expandable: true },
];

grid.expandTemplate = (row) => html`
  <p><strong>Notes for ${row.cells.name}</strong></p>
  <p>${row.cells.notes ?? 'No notes recorded.'}</p>
`;

grid.expandedRowIds = [];
grid.addEventListener('civ-row-expand', (e) => {
  const { rowId, expanded } = e.detail;
  grid.expandedRowIds = expanded
    ? [...grid.expandedRowIds, rowId]
    : grid.expandedRowIds.filter((id) => id !== rowId);
});
```

**The detail row** is a new `<tr>` rendered immediately below the data
row, with a single `<td>` spanning all visible columns. The td gets a
tinted background (`--civ-color-base-lightest`) so the detail reads as
a connected region under its parent.

**Nesting a `civ-data-grid` inside `expandTemplate`** is supported and
common for master-detail data hierarchies (e.g. applications → their
supporting documents). See the "Expandable rows → nested civ-data-grid"
story for the pattern.

**Disabled rows** with `expandable: true` get a disabled chevron button
with the same opacity treatment as the rest of the row, and the toggle does
not dispatch `civ-row-expand`.

**Keyboard.** Unlike row activation, expand IS keyboard-accessible: the
chevron is a real `<button>` with `aria-expanded` and `aria-controls`
pointing at the detail `<td>`'s id. Tab focuses it, Enter/Space toggles.

## Inline cell editing

Mark columns as `editable: true` to make their cells click-to-edit.
The grid swaps the cell text for an input on click and commits on
Enter / blur / click-outside. Escape cancels.

```ts
grid.columns = [
  { key: 'name', header: 'Name', editable: true },
  {
    key: 'role',
    header: 'Role',
    editable: true,
    inputType: 'select',
    options: [
      { value: 'admin', label: 'Admin' },
      { value: 'editor', label: 'Editor' },
      { value: 'viewer', label: 'Viewer' },
    ],
  },
  {
    key: 'quota',
    header: 'Quota',
    editable: true,
    inputType: 'number',
    validate: (v) => (typeof v === 'number' && v >= 0 ? null : 'Must be ≥ 0'),
  },
];

grid.addEventListener('civ-cell-edit-commit', (e) => {
  const { rowId, columnKey, value } = e.detail;
  // The grid never mutates rows directly — update your source of truth.
  rows = rows.map((r) =>
    r.id === rowId ? { ...r, cells: { ...r.cells, [columnKey]: value } } : r,
  );
  grid.rows = rows;
});
```

**Input types**: `'text'` (default), `'number'` (commits parsed to `Number`),
`'select'` (requires `column.options`).

**Validation**: `column.validate(value, row) => string | null`. Returning a
non-empty string blocks `civ-cell-edit-commit`, keeps the cell in edit mode,
and renders the error inline below the input with `role="alert"`.

**Other events**: `civ-cell-edit-start` fires on click; `civ-cell-edit-cancel`
fires on Escape. The grid never mutates `rows`. Consumers reflect the new
value back into their data and reassign `grid.rows`.

**Edit mode and other features**:
- Disabled rows (`row.disabled`) are not editable even when `column.editable` is set.
- Editable cells in an `interactive` grid: clicking the cell starts edit mode but does NOT fire `civ-row-activate`. The cell-click stops propagation.
- Only one cell is in edit mode at a time. Starting an edit on another cell would not be a v1 concern (consumer can blur first).

## Group-by collapsible groups

Set `groupBy` to a column key to group rows by that column's value.
Each group renders a header row spanning all visible columns with a
chevron to toggle expansion.

```ts
grid.groupBy = 'type';
grid.expandedGroups = ['Disability', 'Pension', 'Education'];
grid.groupLabel = (key, rows) => `${key} (${rows.length} application${rows.length === 1 ? '' : 's'})`;

grid.addEventListener('civ-group-toggle', (e) => {
  const { groupKey, expanded } = e.detail;
  grid.expandedGroups = expanded
    ? [...grid.expandedGroups, groupKey]
    : grid.expandedGroups.filter((k) => k !== groupKey);
});
```

**Group order**: groups appear in the order their first member appears
in `rows`. The grid does not re-sort rows for grouping. Pre-sort your
data if you need a specific group order.

**`expandedGroups` semantics**: when undefined (the default), all
groups render expanded. Once you assign an array (even `[]`), you
take control. Only groups whose key is in the array stay expanded.

**`groupLabel` callback**: optional `(groupKey, rows) => string`. When
omitted, the header reads `{groupKey} ({count})`. Use this to render
readable labels for technical keys (`'in-review'` → `'In review (3)'`),
to localize, or to embed aggregates.

**Empty group keys**: rows with `cells[groupBy]` set to `null` or
`undefined` are grouped under the empty-string key (`''`); the header
falls back to the i18n `dataGridGroupNoValue` string (`(no value)`).

**Tip. Hide the redundant grouped column.** When you group by `type`,
every data row's "Type" cell shows the same value as its group header.
You can drop the duplicate column by marking it hidden:

```ts
grid.columns = columns.map((c) =>
  c.key === grid.groupBy ? { ...c, hidden: true } : c
);
```

The grid skips hidden columns (header + body). Same path
`civ-column-visibility` uses.

**Interaction with other features:**
- **Sort + grouping**: a naive `rows.sort(...)` across the whole array
  will fracture groups (same group's rows scatter). When sorting an
  active grouped grid, sort *within each group*:
  ```ts
  function sortWithinGroups(rows, groupBy, sortBy, direction) {
    const groups = new Map();
    for (const r of rows) {
      const k = String(r.cells?.[groupBy] ?? '');
      if (!groups.has(k)) groups.set(k, []);
      groups.get(k).push(r);
    }
    const sorted = [];
    for (const [, groupRows] of groups) {
      groupRows.sort((a, b) => /* compare a.cells[sortBy] / b.cells[sortBy] */);
      sorted.push(...groupRows);
    }
    return sorted;
  }
  ```
  Group boundaries are preserved because group order follows row-array
  insertion order.
- **Selection** is per-row independently. There's no select-all-in-
  group affordance in v1.
- **Expandable rows** work alongside groups: a row inside a group can
  still expand to show its detail panel.
- **Sticky columns**: the group label stays anchored at the leading
  edge of the viewport during horizontal scroll (via `position:
  sticky` on the toggle button) so the group identity is always
  visible alongside the sticky data columns.

## Aggregator footer

Set `column.aggregate` on any column to render an aggregator footer. The grid creates a `<tfoot>` automatically and renders a per-column aggregator cell; columns without an aggregator show empty placeholder cells. When `groupBy` is also set, each expanded group gets its own subtotal row at the bottom of its data rows.

### Built-in aggregators

```ts
{ key: 'amount', header: 'Amount', aggregate: 'sum' }   // sum
{ key: 'amount', header: 'Amount', aggregate: 'avg' }   // average (mean)
{ key: 'id',     header: 'ID',     aggregate: 'count' } // non-empty cells
{ key: 'amount', header: 'Amount', aggregate: 'min' }   // smallest finite value
{ key: 'amount', header: 'Amount', aggregate: 'max' }   // largest finite value
```

`sum` / `avg` / `min` / `max` parse cells via `Number()` and silently drop non-finite values (null, undefined, '', 'N/A', etc.). Matches Google Sheets / Excel. `count` counts cells that are present and non-empty (Excel `COUNTA`, not `COUNT`). `min` / `max` on an empty set return `0` rather than `±Infinity`.

### Custom aggregators (formatting, labels, weighted averages)

For anything beyond a plain number (currency formatting, "Total:" prefixes, weighted averages, ratios) pass a function:

```ts
import { sum, avg, count } from '@civui/data/aggregate';

const cols = [
  { key: 'applicant', header: 'Applicant' },
  {
    key: 'amount',
    header: 'Award amount',
    aggregate: (rows, col) =>
      `Total: $${sum(rows, col).toLocaleString('en-US', { maximumFractionDigits: 0 })}`,
  },
  {
    key: 'days',
    header: 'Days in review',
    aggregate: (rows, col) => `Avg: ${avg(rows, col).toFixed(1)} days`,
  },
];
```

The function receives the row subset for the aggregate context (the whole grid for the footer; the group's rows for a group subtotal) plus the column definition. Return a string or number. The grid renders it verbatim, so the function is the right place to format.

### Per-group subtotals

When `groupBy` is set and any column has `aggregate`, each expanded group renders a subtotal row beneath its data rows. The subtotal row uses the same column layout as a data row but each cell shows its column's aggregator computed from that group's rows. Set `showGroupSubtotals="false"` to keep the grand-total footer but drop the per-group rows.

### Sticky footer

`stickyFooter` pins the `<tfoot>` to the bottom of the scrolling container. Useful for long tables where the totals would otherwise scroll off-screen. The grid is responsible for the sticky positioning of its own cells; the table must sit inside a scrollable element with a constrained height for the effect to engage.

```html
<div style="max-height: 400px; overflow-y: auto;">
  <civ-data-grid sticky-footer caption="Applications"></civ-data-grid>
</div>
```

### Aggregates + filters

The footer aggregates **whatever rows the consumer hands to the grid**. There's no internal filter / paginate step. When you use `applyGridFilters` client-side, set `grid.rows` to the filtered set and aggregates recompute automatically:

```ts
grid.addEventListener('civ-filter-change', (e) => {
  grid.filters = e.detail.filters;
  grid.rows = applyGridFilters(allRows, cols, grid.filters);
  // Footer + subtotals recompute from the new rows on the next render.
});
```

For server-side data, the consumer can also return per-column server-computed aggregates and surface them via custom function aggregators that ignore the `rows` arg and read from a separate state slot. Useful when totals are computed across the full dataset, not just the current page.

## Per-column filtering

Set `column.filter` on any column to render a filter input in a row beneath the column headers. Three variants ship today:

- `text`: case-insensitive substring match (one `<input type="text">`).
- `select`: exact-match dropdown built from the supplied options.
- `number-range`: pair of min / max number inputs.

The grid is consumer-controlled: every keystroke / change dispatches `civ-filter-change` with the new full filter state, and the consumer applies the filter to its row data. For client-side filtering, the `applyGridFilters` utility from `@civui/data/filter` is the canonical implementation:

```ts
import { applyGridFilters } from '@civui/data/filter';

const cols = [
  { key: 'name',   header: 'Name',   sortable: true, filter: { type: 'text', placeholder: 'Search…' } },
  { key: 'status', header: 'Status', filter: { type: 'select', options: [
    { value: 'active',  label: 'Active' },
    { value: 'pending', label: 'Pending' },
  ]}},
  { key: 'amount', header: 'Amount', align: 'numeric', filter: { type: 'number-range' } },
];

const allRows = await fetchRows();
grid.columns = cols;
grid.rows = allRows;
grid.filters = {};

grid.addEventListener('civ-filter-change', (e) => {
  const { filters } = e.detail;
  grid.filters = filters;
  grid.rows = applyGridFilters(allRows, cols, filters);
});
```

### Filter row layout

Cells in the filter row line up one-to-one with the header cells above:

- Selection, expand, and row-actions columns render an empty placeholder cell.
- Columns without a `filter` config also render an empty placeholder cell (so the visual grid stays aligned).
- The cells that DO carry filters render the input directly inside the cell. No extra label / hint chrome, since the column header above already provides the label. The input's `aria-label` reads `"Filter <column>"` for screen readers.

### Filter semantics (`applyGridFilters`)

- **`text`:** `String(cell).toLowerCase().includes(value.toLowerCase())`. Empty input is a no-op.
- **`select`:** `String(cell) === value`. Empty option (`value=""`) is the "All" no-op.
- **`number-range`:** `min <= Number(cell) <= max` (both bounds inclusive, both optional). Cells whose value is `null` / `undefined` / `""` or doesn't parse to a finite number are **excluded** from any active range. Matches Google Sheets / Excel behavior so blank cells don't sneak into "≥ 0" filters.
- Multiple active filters **AND** together. A row must satisfy every filter to appear.

### Server-side filtering

`civ-filter-change` is a target-state event. `e.detail.filters` is the full filter map, not a diff. Hand it to your query API:

```ts
grid.addEventListener('civ-filter-change', async (e) => {
  grid.filters = e.detail.filters;
  grid.loading = true;
  const { rows, total } = await api.search({ filters: grid.filters, page: 1 });
  grid.rows = rows;
  grid.loading = false;
});
```

`e.detail.columnKey` is also provided when you need to know which column the user touched. Useful for analytics or for debouncing per-column inputs.

### Filters + sort + grouping

The three composite features are orthogonal:

- **Filters apply first** (when using `applyGridFilters` client-side), then the consumer sorts and groups the filtered set. The grid renders whatever rows you hand it.
- The filter row sits in `<thead>` underneath the header row, above the body. Sort indicators on column headers continue to function normally above their filter inputs.
- `groupBy` groups the rows that survive filtering. Empty groups disappear naturally.

### Filters + keyboard navigation

With `keyboardNav` on, filter cells participate in cell navigation. Pressing <kbd>Enter</kbd> on a focused filter cell focuses the inner input so the user can start typing; <kbd>Esc</kbd> returns focus to the cell so arrow keys resume.

## Keyboard navigation {#keyboard-navigation}

Set `keyboardNav` to promote the table to `role="grid"` with 2D arrow-key navigation per the [WAI-ARIA Grid Pattern](https://www.w3.org/WAI/ARIA/apg/patterns/grid/). Opt in to a single tab stop with arrow-key cell navigation, instead of the default tab-through-every-control behavior. The grid becomes a **single tab stop**. Inner controls (sort buttons, expand toggles, action menus, edit triggers, checkboxes) drop to `tabindex="-1"` so Tab doesn't walk through every cell. Once focus is on the grid, arrow keys move between cells and Enter / Space activate the cell's primary control.

```html
<civ-data-grid
  caption="Applications"
  keyboard-nav
  bordered
></civ-data-grid>
```

### Key map

| Key | Action |
|---|---|
| <kbd>Tab</kbd> | Focus the grid (the cell at the current roving-tabindex position). Tab again to leave. |
| <kbd>←</kbd> <kbd>→</kbd> | Move one cell left / right within the current row. |
| <kbd>↑</kbd> <kbd>↓</kbd> | Move one row up / down (header ↔ body included). |
| <kbd>Home</kbd> | First cell in the current row. |
| <kbd>End</kbd> | Last cell in the current row. |
| <kbd>Ctrl</kbd>+<kbd>Home</kbd> | First cell of the grid (header row, first column). |
| <kbd>Ctrl</kbd>+<kbd>End</kbd> | Last cell of the last body row. |
| <kbd>PageDown</kbd> / <kbd>PageUp</kbd> | Jump ±10 rows (clamped to the grid). |
| <kbd>Enter</kbd> / <kbd>Space</kbd> | Activate the cell's primary control. Sort the column, toggle the checkbox, open the action menu, start the edit, etc. On a plain cell of an `interactive` row, fires `civ-row-activate`. |
| <kbd>F2</kbd> | Start edit on an editable cell (Excel convention). No-op on other cells. |
| <kbd>Shift</kbd>+<kbd>Enter</kbd> / <kbd>Shift</kbd>+<kbd>Space</kbd> | When `multiSort` is on and the focused cell is a sortable header, adds it to (or cycles direction within) the multi-sort stack. The keyboard equivalent of mouse Shift-click. |
| <kbd>Esc</kbd> | Inside an active edit, cancels the edit. From any inner control of a cell, returns focus to the cell (see [Escape returns focus to the cell](#escape-returns-focus-to-the-cell)). |

### Activation model

Enter / Space find the first focusable descendant of the focused cell (`button, input, select, a, [role="button"]`) and click it. That single rule covers:

- Sortable headers → clicks the sort button → fires `civ-sort`.
- Select-all header / row select cells → clicks the checkbox or radio → fires `civ-selection-change`.
- Expand cells → clicks the chevron → fires `civ-row-expand`.
- Actions cells → clicks the kebab trigger → opens the menu.
- Editable cells → clicks the cell trigger button → enters edit mode (focus moves to the input).

When the cell has no focusable descendant **and** the row is `interactive`, Enter / Space fire `civ-row-activate` instead. So master-detail flows have a keyboard path even on rows that don't carry inline actions.

### Focus after edit

When `keyboardNav` is on and the user commits or cancels an edit, focus returns to the cell (not to `<body>`) so they can keep arrow-keying through data without dropping out of the grid.

### Column memory across single-cell rows

When `↑` / `↓` transit a row that only has one cell (a **group header**, an **expanded detail row**) focus collapses to col 0 (the only cell), but the user's preferred column is remembered. The next vertical move that reaches a multi-cell row restores their column. Matches Excel / Google Sheets behavior.

```text
   Name   Type   Status
─────────────────────────────
        ▼ Type X
   A       X      On      ← user navigates here, then ↓
        ▼ Type Y          ← group row collapses to col 0
   B       Y      Off     ← col 2 ("Status") restored
   C       Y      On
```

Explicit horizontal navigation (`←` / `→`, `Home` / `End`, mouse click) updates the memory; transit through single-cell rows does not.

### Escape returns focus to the cell

When focus has moved off the cell onto an inner control. E.g. a sort button after clicking it, or the row-actions kebab after closing its menu. Pressing `Esc` returns focus to the cell so arrow keys resume cell navigation. This is the WAI-ARIA "actionable cell" pattern.

When a `<civ-menu>` is currently open inside a cell, the grid intentionally bails on `Esc` so the menu can close itself and restore focus to its trigger; the next `Esc` (now with the menu closed) walks the focus from the trigger back out to the cell. Two presses → cell focused → arrows resume.

Arrow keys also continue to work directly from an inner control. The keydown handler picks up the closest cell on every event, so the user doesn't have to press `Esc` first to navigate vertically out of a sort button.

### Screen reader mode

`role="grid"` flips JAWS / NVDA / VoiceOver into "applications" or "forms" mode while focus is inside the grid: the standard table-reading shortcuts (T to jump tables, Ctrl+Alt+Arrow to move by row / column) are suppressed in favor of the application's own arrow-key handling. Which is now the grid's. This is the intended WAI-ARIA Grid behavior. **Leave `keyboardNav` off for simple readable tables** so SR users keep their normal table-reading affordances.

### When to use it

- ✅ Admin / back-office screens with sortable + editable + selectable + action-menu cells where Tab would walk through dozens of controls per page.
- ✅ Power-user workflows where a keyboard-first review of many rows is faster than a mouse-driven one.
- ❌ Simple readable tables (no inline editing, no row actions). Leave `keyboardNav` off so screen readers stay in normal table reading mode.

## Export to CSV / JSON

`@civui/data/export` exports a `gridExport` utility. A pure function
that returns a `File` (a named `Blob`) the consumer can hand to an
anchor download, `navigator.share`, or any other Blob-consuming path.
No component, no event dispatch; the data-grid stays focused on
rendering, the export utility stays focused on serialization.

```ts
import { gridExport } from '@civui/data/export';

function downloadCsv(grid: HTMLElement & { rows: GridRow[]; columns: GridColumn[] }) {
  const file = gridExport(grid.rows, grid.columns, {
    format: 'csv',
    filename: 'applications.csv',
  });
  const url = URL.createObjectURL(file);
  const a = document.createElement('a');
  a.href = url;
  a.download = file.name;
  a.click();
  URL.revokeObjectURL(url);
}
```

**Options:**

| Option | Default | Behavior |
|---|---|---|
| `format` | `'csv'` | `'csv'` (RFC 4180 quoting, CRLF line endings) or `'json'` (pretty-printed array of `row.cells` objects) |
| `filename` | `'export.csv'` / `'export.json'` | Name on the returned `File`. MIME type is set automatically (`text/csv` / `application/json`). |
| `includeHidden` | `false` | When `false`, columns with `hidden: true` are omitted. Matches what the user is seeing. Set `true` to export every declared column. |
| `useFormatter` | `false` (CSV only) | When `true`, the column's `formatter` callback produces the cell's text. Most formatters return `TemplateResult` which doesn't stringify usefully. Only enable when you've authored plain-value formatters. |
| `selectedRowIds` | (all rows) | When provided, only rows whose `id` is in this array are exported. Pairs with `grid.selectedRowIds` for "export selection" flows. |

**CSV quoting** follows RFC 4180: values containing `,`, `"`, or
`\r\n` are wrapped in double quotes with embedded `"` doubled. Plain
values are emitted unquoted. Header cells follow the same rule.

**JSON shape**: an array of `row.cells` subset objects keyed by the
exported `column.key`. Native types are preserved (numbers stay
numbers, booleans stay booleans). Unlike CSV where everything
stringifies.

## Sticky columns

Set `column.sticky: 'start'` to pin a column to the leading edge or
`'end'` to pin it to the trailing edge. Pinned columns stay visible
while the rest of the table scrolls horizontally. Useful for wide
admin tables where the row-identity column (ID / name) needs to stay
anchored.

```ts
grid.columns = [
  { key: 'id', header: 'ID', width: '5rem', sticky: 'start' },
  { key: 'applicant', header: 'Applicant', width: '14rem', sticky: 'start' },
  { key: 'type', header: 'Type', width: '10rem' },
  // …more columns
  { key: 'updated', header: 'Last updated', width: '8rem', sticky: 'end' },
];
grid.responsive = 'scroll';
```

**Constraints:**
- Multi-column sticky requires explicit `width` on every pinned column
  (except the first on each edge). CSS can't compute cumulative offsets
  against `width: auto`. The grid fires a dev-mode warning if violated.
- The boundary cell (the inner edge of the pinned group) gets a thin
  shadow so users see the separation from scrolling content.
- Striped, selected, and hover row backgrounds override the sticky
  default surface color so visual state stays visible through pinned
  cells.

**Composition**: sticky columns work with `responsive="scroll"` (the
intended pairing) and also with the default flow-table mode if the
container is narrower than the table's natural width. Sticky doesn't
interact with `responsive="stacked"`. On mobile the table collapses
to vertical blocks and column-level pinning isn't meaningful.

## Responsive behavior

Two modes via the `responsive` prop:

- **`stacked` (default):** On ≤480px, each row collapses into a vertical block. Each `<td>` renders its column header (from `data-label`) inline next to the value. The USWDS stacked table pattern.
- **`scroll`:** Wraps the table in a horizontally-scrollable region. Rows stay tabular at all viewport widths. Use when columns must stay in row order on small screens.

Both modes preserve the same DOM. CSS does the work, so screen readers always see a proper `<table>`.

## Server-side data

The grid is controlled, so server-side sorting and pagination are natural:

```ts
const grid = document.querySelector('civ-data-grid')!;
const pager = document.querySelector('civ-pagination')!;

let state = { page: 1, pageSize: 25, sortBy: '', sortDirection: 'none' as const };

async function fetchAndRender() {
  grid.loading = true;
  try {
    const { rows, total } = await api.search(state);
    grid.rows = rows;
    pager.totalItems = total;
    pager.page = state.page;
    grid.loading = false;
  } catch (err) {
    grid.errorMessage = err instanceof Error ? err.message : 'Failed to load';
    grid.loading = false;
  }
}

grid.addEventListener('civ-sort', (e) => {
  state.sortBy = e.detail.column;
  state.sortDirection = e.detail.direction;
  state.page = 1; // reset paging when sort changes
  fetchAndRender();
});

pager.addEventListener('civ-page-change', (e) => {
  state.page = e.detail.page;
  state.pageSize = e.detail.pageSize;
  fetchAndRender();
});
```

## Coming in follow-up work

The web component is feature-complete for v1. Two convenience layers remain on the wishlist; pick them up when a consumer explicitly asks:

- **Saved views / presets:** user-saveable named bundles of column visibility + filters + sort. Composes existing primitives; the only new piece is a storage abstraction (consumer-controlled by default).
- **URL state persistence:** sync sort / filters / page to query params for shareable filtered views. Pure utility module; no component changes.

The remaining real work is platform parity:

- **Native iOS / Android implementations:** schema is in place; native bodies are placeholders. This is a dedicated implementation pass, not a feature pass. See `.claude/rules/audit-debt.md`.

Out of scope for this component (AG-Grid-Enterprise-tier features that government admin tools rarely need): column resizing, column reordering, row drag-reordering, virtual scrolling, cell-range selection, clipboard, header tooltips, truncation tooltips. Open an issue if a real consumer needs one.
