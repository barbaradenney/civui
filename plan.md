# Plan: Unified Web Components for Web + Mobile

## Goal
Replace `@civui/react-native` with a single set of Lit web components that work on both web browsers and mobile apps (via WebView-based runtimes like Capacitor). One codebase, one component library, one design system.

## Why This Works Now
- CivUI already uses **Light DOM** — no Shadow DOM encapsulation means standard CSS and DOM APIs work everywhere a WebView exists.
- **Capacitor** (by Ionic) wraps any web app in a native shell with full access to native APIs (camera, filesystem, biometrics, push notifications). It runs a real WebView, so Lit components render identically to the browser.
- The React Native package is a **full re-implementation** of every component (13 components, ~3000 lines of duplicated logic) that must be manually kept in sync. Eliminating this removes an entire maintenance surface.
- Web components are a W3C standard supported by all modern WebViews (iOS 15.4+, Android 5+).

## What Gets Replaced

| Current (React Native) | Unified (Web Components) |
|---|---|
| 13 React Native form components | Same 15 Lit web components used on web |
| React Context theme provider | CSS custom properties (already in tokens) |
| Hardcoded token values in `tokens.ts` | Single `tokens.css` from `@civui/tokens` |
| `react-native` + `react-native-web` | Capacitor + standard web platform |
| Jest + `@testing-library/react-native` | Same Vitest + jsdom tests |
| Separate Storybook (port 6007) | Same Storybook (port 6006) |

## Implementation Steps

### Phase 1: Mobile Adaptation Layer (in `@civui/core`)

**1.1 Add responsive breakpoints to Tailwind config**
- Add `sm` (640px), `md` (768px), `lg` (1024px) breakpoints with `civ-` prefix
- These enable responsive layouts without changing existing components

**1.2 Create mobile utility module (`packages/core/src/mobile/`)**
- `platform.ts` — detect if running in Capacitor vs plain browser (`window.Capacitor` check)
- `viewport.ts` — safe area inset helpers (CSS `env(safe-area-inset-*)` wrappers)
- `haptics.ts` — optional haptic feedback wrapper (calls Capacitor Haptics plugin if available, no-op otherwise)
- These are lightweight utilities, not framework dependencies

**1.3 Add touch-optimized styles**
- Increase tap targets to 44×44px minimum on touch devices via `@media (pointer: coarse)`
- Add momentum scrolling (`-webkit-overflow-scrolling: touch`) where needed
- Add touch-action hints to interactive elements

**1.4 Adapt components that behave differently on mobile**
- `civ-select`: On mobile, optionally use native `<select>` for OS-level picker (better UX for long lists). Add `native` boolean attribute.
- `civ-date-picker`: Ensure the calendar popover is full-screen on small viewports (already possible with CSS)
- `civ-combobox`: Ensure virtual keyboard doesn't obscure the dropdown
- `civ-file-upload`: Wire into Capacitor Camera/Filesystem plugins when available

### Phase 2: Capacitor Shell (`packages/mobile/`)

**2.1 Create new `@civui/mobile` package**
- Replace `@civui/react-native` in the monorepo
- Contains Capacitor project config, not component code
- Structure:
  ```
  packages/mobile/
  ├── package.json
  ├── capacitor.config.ts
  ├── ios/                  # Generated by Capacitor
  ├── android/              # Generated by Capacitor
  └── src/
      ├── index.html        # Entry point loading web components
      ├── app.ts            # App initialization + Capacitor plugin setup
      └── plugins/          # Thin wrappers around native capabilities
          ├── camera.ts
          ├── haptics.ts
          ├── storage.ts
          └── push.ts
  ```

**2.2 Plugin integration pattern**
- Each plugin wrapper exports a simple async function
- Components detect Capacitor at runtime and use enhanced features when available
- No hard dependency — components work without Capacitor (progressive enhancement)

### Phase 3: Migration & Deprecation

**3.1 Create migration guide**
- Map every React Native component prop to its web component equivalent
- Map React Native events (`onChange`, `onInput`) to custom events (`civ-change`, `civ-input`)
- Map React Native accessibility props to ARIA attributes
- Provide before/after code examples

**3.2 Feature parity checklist**
Verify each RN component's behavior is covered:

| RN Component | Web Component | Gap? |
|---|---|---|
| `TextInput` | `civ-text-input` | None — same props |
| `Textarea` | `civ-textarea` | None |
| `Select` | `civ-select` | Add `native` prop for mobile OS picker |
| `Checkbox` | `civ-checkbox` | None |
| `RadioGroup` | `civ-radio` (group) | None |
| `Toggle` | `civ-toggle` | None |
| `SegmentedControl` | `civ-segmented-control` | None |
| `CheckboxGroup` | `civ-checkbox` (group) | None |
| `Combobox` | `civ-combobox` | None |
| `FormGroup` | `civ-form-group` | None |
| `Fieldset` | `civ-fieldset` | None |
| `MemorableDate` | `civ-memorable-date` | None |
| `DatePicker` | `civ-date-picker` | None |
| `CivThemeProvider` | CSS custom properties | Already works — tokens.css |
| `useForm` hook | `civ-form` element | Already exists |
| `useAnalytics` hook | `sendAnalytics()` method | Already in CivBaseElement |

**3.3 Deprecate and remove `@civui/react-native`**
- Mark package as deprecated in package.json
- Remove from turbo.json pipeline
- Remove from pnpm workspace
- Delete the package directory

### Phase 4: Testing & Storybook

**4.1 Mobile viewport testing**
- Add Storybook viewport addon with mobile presets (iPhone SE, iPhone 14, Pixel 7)
- Add visual regression tests at mobile breakpoints

**4.2 Touch interaction tests**
- Add `pointer: coarse` simulation in test setup
- Test tap target sizes meet 44×44px minimum
- Test that select/combobox dropdowns are usable on small screens

**4.3 Capacitor test harness**
- Create a simple test app in `packages/mobile/` that exercises all components
- Test on iOS Simulator and Android Emulator

## What This Does NOT Include
- **No React wrapper package** — consumers who need React integration can use `@lit/react` (existing Lit project, not our code to maintain)
- **No offline-first/PWA** — that's an app concern, not a design system concern
- **No navigation/routing** — that's app-level, Capacitor or the consumer handles it
- **No state management replacement** — `useForm` was a React hook; `civ-form` already handles this as a web component

## Build Order Impact
Current: `tokens → core → forms → react-native`
New: `tokens → core → forms → mobile` (mobile is a shell, not component code — faster builds)

## Risk Assessment
- **Low risk**: All 15 web components already exist and are tested (422+ tests). We're not rewriting components — we're removing the duplicate RN set.
- **Medium risk**: Mobile-specific UX patterns (native select picker, safe area insets) need design validation on real devices.
- **Managed risk**: Capacitor's WebView quality on older Android devices (pre-Chrome 80) could have rendering gaps. Set minimum Android WebView version in Capacitor config.
