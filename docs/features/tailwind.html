Tailwind CSS Architecture

How CivUI uses Tailwind with the civ- prefix and @apply for consistent, conflict-free component styling.

The civ- Prefix

Every Tailwind utility in CivUI uses the civ- prefix. Government websites frequently ship with existing CSS frameworks — Tailwind, Bootstrap, USWDS — and bare utility classes like p-4 or text-sm would collide with those. The prefix guarantees that CivUI styles never interfere with the host page's CSS, no matter what else is loaded.

civ-p-4
civ-text-error
civ-border-primary
civ-bg-white
civ-font-bold
civ-rounded
civ-w-full
civ-block

This means a government agency can drop CivUI into an existing site that already uses unprefixed Tailwind or Bootstrap without any class name conflicts. Each civ- utility maps one-to-one with its standard Tailwind counterpart — civ-p-4 does exactly what p-4 does, but in its own namespace.

Component Classes with @apply

Rather than scattering 10-15 utility classes across every component template, CivUI defines semantic component classes like .civ-input, .civ-label, and .civ-btn. Each class bundles its Tailwind utilities via @apply, keeping templates clean and styling consistent across all components.

Component Class @apply Utilities Purpose
.civ-label civ-block civ-mb-0 civ-font-bold
+ font-size: var(--civ-typography-fontSize-base)
+ color: var(--civ-color-base-darkest)		 Bold label text in the darkest base color, displayed as a block element
.civ-input civ-block civ-w-full civ-border civ-rounded civ-px-2 civ-py-1.5 civ-text-base civ-font-sans civ-text-base-darkest civ-bg-white civ-border-base-light Full-width block input with border, rounded corners, horizontal and vertical padding, system font
.civ-btn civ-inline-flex civ-items-center civ-justify-center civ-gap-2 civ-font-sans civ-font-bold civ-text-base civ-rounded civ-px-5 civ-py-3 civ-border civ-cursor-pointer civ-transition-colors Inline-flex button with centered content, icon gap, bold font, generous padding, pointer cursor, color transitions
.civ-check-tile civ-relative civ-border civ-rounded civ-border-base-light civ-cursor-pointer Tile-style checkbox or radio with border and pointer cursor, becomes highlighted when checked
.civ-alert civ-p-4 civ-mb-4 civ-border-s-4 Alert container with padding, bottom margin, and a left accent border for status indication

The benefit: a component template uses class="civ-input" instead of listing 11 separate utility classes. This makes templates readable, keeps styling consistent across all instances of each element type, and creates a single place to update the look of every input, button, or label in the system.

Design Token Integration

Tailwind's color, spacing, and typography values in CivUI are not hardcoded. They come from CSS custom properties generated by the token build pipeline. The token system is the single source of truth.

For example, when you write civ-text-primary, Tailwind resolves it to color: var(--civ-color-primary-DEFAULT). That CSS variable is generated by the token build from the design token color.primary.DEFAULT defined in the W3C DTCG JSON files.

The full chain:

Token JSON → Build Script → CSS Variables → Tailwind Config → Component Classes 
/* Token JSON (W3C DTCG format) */
{
  "color": {
    "primary": {
      "DEFAULT": { "$value": "#005ea2", "$type": "color" }
    }
  }
}

/* Generated CSS variable */
--civ-color-primary-DEFAULT: #005ea2;

/* Tailwind config maps to the variable */
primary: { DEFAULT: 'var(--civ-color-primary-DEFAULT)' }

/* Usage in @apply */
.civ-btn--primary {
  @apply civ-bg-primary civ-text-white civ-border-primary;
}

This means changing a token value in the JSON source automatically propagates through the CSS variables, into Tailwind, and out to every component that uses it. Dark mode works the same way: a separate color-dark.tokens.json generates @media (prefers-color-scheme: dark) overrides for the same CSS variables.

State Selectors

Component states are handled via CSS attribute selectors and pseudo-classes rather than JavaScript class toggling. The component sets the appropriate ARIA attribute or native state, and CSS handles the visual change. This keeps styling declarative and ensures that the visual state always matches the accessibility state.

Selector Styles Applied What it does
.civ-input[aria-invalid="true"] civ-border-error civ-border-s-4 Red border with thicker left accent on invalid inputs
.civ-input:disabled civ-opacity-50 civ-cursor-not-allowed civ-bg-base-lightest Faded appearance with not-allowed cursor on disabled inputs
.civ-check-tile:has(:checked) civ-border-primary civ-bg-primary-lightest Blue border and light blue background on selected tile checkboxes/radios
.civ-toggle-track[aria-checked="true"] civ-bg-primary civ-border-primary Solid blue track when toggle is switched on
.civ-segment-btn[aria-checked="true"] civ-bg-primary civ-text-white civ-border-primary Blue background with white text for the active segment
.civ-dropzone[data-dragging] civ-border-primary civ-bg-primary-lightest Blue highlight when a file is dragged over the upload zone

Because styles are tied to ARIA attributes and native pseudo-classes, a component cannot look "selected" without actually being selected for assistive technology. The visual and accessible states are always in sync.

Design Principles

  • Border-radius on interactive elements only. Buttons, inputs, tiles, and dropdowns get rounded corners. Static containers and layout elements do not.
  • Underlines only on link-styled buttons. When a <civ-button> renders as an anchor (a.civ-btn), it gets an underline to signal navigation. Action buttons (button.civ-btn) do not.
  • Error states use color + border + text. Color alone is never sufficient. Invalid inputs get a red border, a thicker left accent (civ-border-s-4), and a text error message with role="alert".
  • Focus uses focus-visible:civ-focus-ring. The focus ring only appears on keyboard navigation (not mouse clicks), using the :focus-visible pseudo-class rather than :focus.
  • Logical properties for RTL safety. Utilities like civ-border-s-4, civ-rounded-s, and civ-me-2 use CSS logical properties so layouts work correctly in right-to-left languages without any code changes.

Usage

Developers using CivUI components do not need to know Tailwind or apply any utility classes. The component handles all styling internally: 

<civ-text-input
  label="Email address"
  name="email"
  type="email"
  required
  hint="For example: name@agency.gov"
></civ-text-input>

This single custom element renders a complete, styled input with label, hint text, and validation — all using the semantic component classes internally. The rendered output includes elements with class="civ-label", class="civ-hint", and class="civ-input", but the consumer never writes those classes directly.

For developers building custom layouts or extending CivUI, the component classes are available to use directly: 

<!-- Using CivUI component classes in custom markup -->
<label class="civ-label">Custom label</label>
<span class="civ-hint">Helper text</span>
<input class="civ-input focus-visible:civ-focus-ring" type="text" />
<button class="civ-btn civ-btn--primary focus-visible:civ-focus-ring">Submit</button>
