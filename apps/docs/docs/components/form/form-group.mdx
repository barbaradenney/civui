# Form Headings & Chrome

Every CivUI form control renders its own label / legend / hint / error / required-indicator chrome from its own props. There is no separate wrapper component for single inputs — set `label="..."` (or `legend="..."` on group components) directly on the control.

```html
<!-- Single input -->
<civ-text-input
  label="Email address"
  name="email"
  type="email"
  hint="Work email preferred"
  required
></civ-text-input>

<!-- Group control -->
<civ-radio-group legend="Preferred contact method" name="contact" required>
  <civ-radio value="email" label="Email"></civ-radio>
  <civ-radio value="phone" label="Phone"></civ-radio>
</civ-radio-group>
```

The only wrapper that remains is [`<civ-form-fieldset>`](/components/form/fieldset), reserved for **multi-field grouping** — a custom block of several unrelated controls under one section heading (e.g. a hand-rolled address composed from individual `civ-text-input`s).

## Self-contained components

These components render their own labels and do **not** need wrappers:

- `<civ-checkbox>` and `<civ-toggle>` — inline labels
- `<civ-address>`, `<civ-name>`, `<civ-direct-deposit>`, `<civ-signature>` — compound components that render their own internal chrome and sub-control labels
- All self-contained group components (`<civ-radio-group>`, `<civ-checkbox-group>`, `<civ-segmented-control>`, `<civ-yes-no>`, `<civ-memorable-date>`, `<civ-date-range-picker>`) — pass `legend` directly; do NOT wrap in `<civ-form-fieldset>` (you'd get nested fieldsets with double legends)

## Heading hierarchy

Government forms commonly follow the **VA.gov / GOV.UK "one question per page" pattern**, where the main question on each page is *also* the page's `<h1>`. This:

- Lets screen-reader users navigate forms by heading.
- Satisfies WCAG 2.4.6 (Headings and Labels) and 2.4.10 (Section Headings).

The `heading-level` prop adds `role="heading"` + `aria-level=N` to the rendered `<label>`/`<legend>`. The native element is preserved so click-to-focus and `for`/`aria-labelledby` still work — heading promotion is purely an accessibility-tree affordance. The `size` prop is independent: visual prominence and screen-reader semantics tune separately.

**Rules:**

- **Exactly one `h1` per page.** On a single-question page, set `heading-level="1"` on the field that asks the question.
- On a multi-question page, the page or step uses its own `<h1>` and each fieldset uses `heading-level="2"` or `"3"`.
- Pair a higher `heading-level` with a larger `size` so the visual hierarchy matches the semantic hierarchy.

**Single-question page (the question is the h1):**

```html
<civ-text-input
  label="What is your email address?"
  name="email"
  type="email"
  heading-level="1"
  size="xl"
  required
></civ-text-input>
```

**Multi-question page with a fieldset as h2:**

```html
<h1 class="civ-heading-xl">Your contact information</h1>

<civ-radio-group
  legend="Preferred contact method"
  name="contact"
  heading-level="2"
  size="md"
  required
>
  <civ-radio value="email" label="Email"></civ-radio>
  <civ-radio value="phone" label="Phone"></civ-radio>
</civ-radio-group>
```

The same `heading-level` and `size` props are available on every form control, `<civ-fieldset>`, `<civ-form-fieldset>`, `<civ-repeater>`, and every compound component (`<civ-name>`, `<civ-address>`, `<civ-direct-deposit>`, `<civ-partnership-history>`, `<civ-race-ethnicity>`, `<civ-relationship>`, `<civ-service-history>`, `<civ-signature>`).
