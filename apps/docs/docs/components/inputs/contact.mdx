# Contact

`<civ-phone>` `<civ-email>`

Pre-configured text-input wrappers for the two contact fields most
gov forms collect: phone number and email address. Each component
ships with the right mask, validator, `inputmode`, default label, and
hint. So the consumer writes a single tag and gets the canonical
treatment.

Both extend [`civ-text-input`](./text-input.mdx) and dispatch the
same events (`civ-input`, `civ-change` with `{ value: string }`).

## When to use

| Field | Component | Why |
| --- | --- | --- |
| US phone number | `<civ-phone>` | `(###) ###-####` mask on blur, `tel` keyboard, plain-language hint |
| International phone number | `<civ-phone international>` | Single field, no mask, `phoneIntl` validator (7-15 digits with optional `+` or `00` prefix) |
| Email address | `<civ-email>` | `email` keyboard, RFC-compliant validator, default `autocomplete="email"` |

## Phone (`<civ-phone>`)

Pre-configured input for phone numbers with automatic formatting,
masking, and validation. Supports US and international formats.

Default label: "Phone number"

| Prop | Type | Default | Description |
|------|------|---------|-------------|
| `international` | `boolean` | `false` | Enable international phone number format (skips the US `(###) ###-####` mask and switches to the `phoneIntl` validator) |

### International format

<StoryEmbed id="forms-inputs-text-input--phone-international" />

International mode is **deliberately a single input**. No country picker, no flag, no separate `+CC` field. Three reasons:

- **Autofill compatibility.** `autocomplete="tel"` matches a browser-saved phone in one shot. The split `tel-country-code` + `tel-national` pattern has spotty cross-browser support (Safari and Firefox often drop the country code), so most users would lose their saved value.
- **Permissive client-side validation.** The `phoneIntl` validator accepts `+44 7700 900123`, `0044 7700 900123`, and `7700 900123`. Anything 7–15 digits with optional `+` or `00` prefix. Strict E.164 normalization is a **server-side** job using libphonenumber; the client just checks "looks like a phone number."
- **No mask.** Phone-number formats vary too much across countries to assume a fixed shape. Users type freely; the validator runs on blur.

### Usage

```html
<!-- US phone number -->
<civ-phone label="Phone number" name="phone" required></civ-phone>

<!-- International phone number -->
<civ-phone label="Phone number" name="phone" international required></civ-phone>
```

## Email (`<civ-email>`)

Pre-configured input for email addresses with built-in validation and
plain-language labeling. Sets `type="email"`, `inputmode="email"`, and
the `email` validator. So blur surfaces "Enter a valid email address"
without a server round-trip.

Default label: "Email address"

<StoryEmbed id="forms-inputs-text-input--email" />

### Usage

```html
<civ-email label="Email address" name="email" required></civ-email>
```

## Accessibility

Both components inherit the [`civ-text-input`](./text-input.mdx)
accessibility contract:

- Label is required (the default fallback covers consumers who forget,
  but plain-language consumer-supplied labels are preferred).
- Validation errors fire on blur and announce via `role="alert"` on the
  rendered `.civ-error-text` span.
- `aria-describedby` wires up the hint and any error message to the
  inner `<input>` so screen readers announce them in context.

## Composition

Both components emit raw, unmasked values via `civ-input` /
`civ-change`. The consumer always sees `1234567890` (digits only),
never `(123) 456-7890`. That means server-side handlers stay simple
and the same value round-trips back into the input cleanly.

```html
<civ-form>
  <civ-fieldset legend="Contact information">
    <civ-email name="email" required></civ-email>
    <civ-phone name="phone" required></civ-phone>
  </civ-fieldset>
</civ-form>
```
