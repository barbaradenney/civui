`<civ-partnership-history>`

Compound component for a single long-term partner relationship — covers marriage, civil union, registered domestic partnership, common-law / cohabitation, and other long-term partnerships. Renders the partner's name, the relationship's start date and location, current status, and a conditional end date. Designed to be used inside a `civ-repeater` for multiple partnership entries.

The component adapts its status vocabulary by the selected partnership type:
- **Marriage** (or no type shown): *Currently married / Divorced / Widowed / Annulled*
- **Civil union, domestic partnership, cohabitation, other**: *Currently together / Relationship ended / Partner has passed away*

## Conditional fields

- **Type-specific field set.** When `show-marriage-type` is on, the dropdown selection drives which middle-section fields render: marriage shows date + city + state; civil union shows registration date + jurisdiction; cohabitation shows start date + state + optional description; other shows approximate date + description.
- **End date.** Only renders when status is **not** the "current/together" option. The fieldset legend adapts to the chosen status (*Date marriage ended*, *Date the relationship ended*, *Date of their passing*).

## Usage

```html
<!-- Marriage-only form -->
<civ-partnership-history
  legend="About this marriage"
  name="entry"
></civ-partnership-history>

<!-- Inclusive partnership form (auto-adapts vocabulary by type) -->
<civ-partnership-history name="entry" show-marriage-type></civ-partnership-history>
```

## Splitting across multiple pages

The full single-page form can run long when the marriage type is shown
and a status with end-date is captured. For applications that prefer a
"one question per page" feel, use the `step` prop to render one section
at a time. Each step keeps the same `MarriageValue` serialization, so
the value threads through unchanged.

| Step | Renders |
|------|---------|
| `who` | Partner's name and (when `show-marriage-type`) the type selector |
| `details` | Type-specific date / location / jurisdiction / description fields |
| `status` | Current status radios and the conditional end-date |

When `legend` is omitted, each step uses a step-specific default
("About your partner" / "Partnership details" / "Partnership status").

```html
<civ-form-step>
  <div data-step-label="About your partner">
    <civ-partnership-history step="who" name="entry" show-marriage-type></civ-partnership-history>
  </div>
  <div data-step-label="Partnership details">
    <civ-partnership-history step="details" name="entry" show-marriage-type></civ-partnership-history>
  </div>
  <div data-step-label="Current status">
    <civ-partnership-history step="status" name="entry" show-marriage-type></civ-partnership-history>
  </div>
</civ-form-step>
```

## Multiple entries via repeater

For applications that allow multiple partnership entries (e.g. a full
marriage history), wrap the compound in `<civ-repeater>` with
`mode="form-steps"`. Each row opens a multi-step add/edit flow that
mirrors the three-page split above:

```html
<civ-repeater
  legend="Partnership history"
  name="partnerships"
  item-label="partnership"
  mode="form-steps"
  min="0"
  max="10"
>
  <div data-step-label="Partner">
    <civ-partnership-history step="who" name="entry" show-marriage-type></civ-partnership-history>
  </div>
  <div data-step-label="Details">
    <civ-partnership-history step="details" name="entry" show-marriage-type></civ-partnership-history>
  </div>
  <div data-step-label="Status">
    <civ-partnership-history step="status" name="entry" show-marriage-type></civ-partnership-history>
  </div>
</civ-repeater>
```

<StoryEmbed id="forms-compound-partnership-history--in-repeater" />

Because each `step` instance keeps its own state, thread the value
between them by listening for `civ-input` on the wrapping repeater
and copying the source's `marriageValue` to its peers. The
[Storybook example](pathname:///civui/storybook/?path=/story/forms-compound-partnership-history--in-repeater)
shows the listener in full.
